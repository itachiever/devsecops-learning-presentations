CBOM Data Fields

CERT-In Guidelines Simplified — Field by Field

✦ Understanding What Data a CBOM Record Contains ✦

📦 Component Info
🔐 Crypto Details

A CBOM is only as useful as the data fields it contains. CERT-In (India's national cybersecurity agency) defines a structured set of fields every CBOM record must include. This episode walks through each field — what it means, why it matters, and what it looks like in a real system.

📌 CERT-In = Computer Emergency Response Team – India. Part of the Ministry of Electronics & IT (MeitY). Defines cybersecurity standards and incident response guidelines for Indian organisations and beyond.

🏷️ Field Group 1 of 4 · Slide 2 of 6

Basic Component Information

📖 These fields form the core identification layer of a CBOM record. They answer the fundamental question: "What cryptographic component is this?" — giving every entry a unique, traceable identity across your entire software estate.

🏷️ Identification Fields — required for every CBOM entry
📛 Component Name Required
The official, unambiguous name of the cryptographic component, library, or tool. Must match the vendor's published name exactly to enable cross-referencing with CVE databases and vulnerability feeds.
Example: OpenSSL  ·  BouncyCastle  ·  libsodium  ·  Windows CNG
🔢 Version Required
The specific release version in use. This is critical — the same library at two different versions can have completely different security postures. Vulnerability databases are version-specific. "Latest" is not acceptable — an exact version string is required.
Example: OpenSSL 3.0.2  ·  BouncyCastle 1.70  ·  libsodium 1.0.18
📝 Description Important
A brief human-readable summary of what this component does and its purpose within the application. Helps auditors and new team members understand the component's role without reading source code.
Example: "TLS library used for HTTPS on payment API gateway"
🔍 Traceability Fields
🏷️ Unique Identifier (PURL / CPE) Required
A standardised machine-readable identifier that uniquely identifies the component globally. Enables automated tooling to match this entry against vulnerability databases without ambiguity.
Example: pkg:generic/openssl@3.0.2
🏛️ Supplier / Vendor Important
The organisation or individual that develops and maintains this component. Used for vendor risk assessment and to know who to contact if a critical vulnerability is discovered.
Example: OpenSSL Software Foundation  ·  Legion of the Bouncy Castle
📄 License Type Optional
The open-source or commercial license governing use of this component. Relevant for compliance, legal review, and understanding redistribution rights in enterprise deployments.
Example: Apache-2.0  ·  MIT  ·  OpenSSL License

💡 Why Version Matters: OpenSSL 3.0.1 was vulnerable to CVE-2022-3602 (Critical). OpenSSL 3.0.7 patched it. Same name — completely different risk profile.

Cryptographic Details

📖 These fields define the security definition layer — the technical specifics of how data is protected. They answer: "Which algorithm protects this data, with what key, over which protocol?" These are the fields auditors scrutinise most closely for compliance (FIPS, PCI-DSS, HIPAA).

⚙️ Algorithm Fields
⚙️ Algorithm Type Required
The category and name of the cryptographic algorithm in use — symmetric, asymmetric, hashing, or signature. CBOM tools flag algorithms against an approved-list (e.g., FIPS 140-3 approved algorithms).
Example: AES-256-GCM  ·  RSA-2048  ·  SHA-256  ·  MD5 ❌ Flagged
🎛️ Algorithm Mode / Padding Important
Even a strong algorithm can be weakened by a bad mode. AES in ECB mode is insecure; AES in GCM mode is strong. CBOM captures the full algorithm configuration, not just the name.
Example: AES/GCM/NoPadding  ·  AES/ECB ❌  ·  RSA/OAEP
🔗 Protocol Fields
🔗 Protocol & Version Required
The security protocol enabling the communication channel and its exact version. Protocol version is as important as algorithm choice — TLS 1.0 is broken; TLS 1.3 is the gold standard.
Example: TLS 1.3  ·  TLS 1.2  ·  TLS 1.0 ❌  ·  SSH-2
🗝️ Key Fields
📏 Key Type & Size Required
The nature of the key (symmetric/asymmetric/session) and its length in bits. Minimum safe sizes: AES ≥ 128-bit, RSA ≥ 2048-bit, ECC ≥ 256-bit. Keys below minimum are flagged Critical.
Example: AES / 256-bit  ·  RSA / 2048-bit  ·  RSA / 1024-bit ❌
📦 Key Storage Location Required
Where the key is stored determines its security level. Hardware Security Modules (HSMs) are the gold standard. Keys in config files or environment variables are flagged as high risk in CBOM.
Example: AWS KMS  ·  HashiCorp Vault  ·  Hardcoded in code ❌
🔄 Key Rotation Policy Important
How frequently the key is rotated and when it was last changed. Stale keys that have never been rotated are a significant risk — CBOM tracks last-rotation date and flags overdue rotations.
Example: Rotated every 90 days — Last: 2025-12-01

Certificate & Metadata Fields

📖 These fields capture trust and verification data — the proof that a component is genuine, issued by a trusted authority, and still within its valid lifetime. Without this data, an organisation cannot manage certificate expiry proactively or demonstrate trust chain integrity to auditors.

📜 Certificate Fields — for TLS/SSL certificate entries

Signature AlgorithmRequiredAlgorithm CA used to sign this certSHA256withRSA  SHA1withRSA ❌
🔏 Hash & Integrity Fields
🔏 Component Hash (Digest) Required
A cryptographic hash of the component file itself — proves the file hasn't been tampered with since it was published. CBOM stores the hash so it can be re-verified at any time.
Example: SHA-256: a3f9b2c1...
🔗 Certificate Fingerprint Important
A short hash of the full certificate used to uniquely identify it in logs, monitoring tools, and alerting systems. Easier to work with than full cert data in automation pipelines.
Example: SHA-256 fingerprint: 4A:7B:C2:...
🗃️ Revocation Status (CRL/OCSP) Important
Whether the certificate has been revoked by its issuing CA before expiry. A revoked cert is immediately untrustworthy. CBOM tracks CRL (Certificate Revocation List) and OCSP status.
Example: OCSP: Good  ·  CRL: Revoked ❌

🎯 Compliance Note: PCI-DSS 4.0 requires organisations to maintain a documented inventory of all certificates in use — this is exactly what CBOM certificate fields deliver.

🗺️ Field Group 4 of 4 · Slide 5 of 6

Usage & Dependency Information

📖 These fields provide operational visibility — they answer: "Where is this component used, who owns it, and what depends on it?" Without usage data, a CBOM is just a list. With it, teams can instantly understand the blast radius of any cryptographic risk.

📍 Location & Ownership Fields
📍 Usage Location Required
Exactly where in the application this component is used — which service, module, function, or layer. This field defines the scope of impact if the component is found to be vulnerable.
Example: Payment Gateway API · /auth/login endpoint · Database encryption module
👤 Ownership (Responsible Team) Required
The team or individual accountable for this component — who must act when a vulnerability is discovered or a cert expires. Enables immediate escalation without any investigation delay.
Example: Platform Security Team  ·  payments-team@company.com
⚙️ Configuration Details Important
Key configuration parameters — cipher suites enabled, TLS versions allowed, key derivation settings. Misconfiguration is as dangerous as a vulnerability. CBOM captures config state at scan time.
Example: TLSv1.3, TLSv1.2 | Ciphers: AES-256-GCM | Key exchange: ECDH

🔗 Dependency Relationships Required
A structured map of which software components use this crypto component — direct dependents and transitive consumers. Enables instant blast-radius analysis when a vulnerability is discovered in the crypto component.
📅 Last Scan Timestamp Required
When this CBOM entry was last generated or verified. Stale CBOM data is dangerous — a component may have been updated without a rescan. CERT-In recommends scanning at every build or deployment.

Summary & Key Takeaways

🏷️

Component Info

Who — Name, version, supplier, identifier. Core identification for every CBOM entry. Powers CVE matching and supply chain tracking.

🔐

Crypto Details

How — Algorithm type, mode, key size, storage, rotation policy, protocol version. Defines the security strength of each component.

📜

Cert & Metadata

Trust — Issuer, expiry, fingerprint, revocation status, hash digest. Proves integrity and manages certificate lifecycle proactively.

🗺️

Usage & Dependencies

Where & Who — Location, owner, config, dependency map, scan timestamp. Delivers operational visibility and blast-radius analysis.

🎯 Unified CBOM Data = Visibility + Compliance + Risk Management: When all four field groups are populated, a CBOM enables security teams to detect weak crypto, track certificate lifecycles, prove regulatory compliance (FIPS, PCI-DSS, HIPAA, CERT-In), and respond to incidents in minutes — not days.

⚙️ CERT-In Standard: Field groups align with India's national CBOM framework — ensuring organisations generating CBOMs meet government cybersecurity requirements for critical infrastructure and regulated industries.

🔄 Living Document: A CBOM is not a one-time report. CERT-In recommends regenerating CBOM at every build, deployment, and dependency update to keep data current and accurate.
