⚠️ The Problem · Slide 2 of 6

Why Containers & Cloud Need CBOM

📖 In traditional server deployments, a team knew which OS and libraries were installed. In container and cloud environments, applications are composed of dozens of images, microservices, and managed cloud services — each bringing their own cryptographic components. Hidden crypto risks multiply at every layer.

🔍 Where Crypto Hides in Modern Applications
  • 🐳
    Container base images — every Docker image inherits its parent image's crypto libraries. A node:18-alpine image includes OpenSSL and TLS libraries you may never have explicitly chosen.
  • 🔗
    Microservice APIs — each service communicates over TLS. In a 50-service architecture, that means 50+ TLS configurations, each with its own certificate and cipher suite choices.
  • ☁️
    Cloud managed services — AWS S3, RDS, DynamoDB — all use encryption at rest and in transit. The encryption algorithm and key management policies are cloud-controlled but still need to be inventoried.
  • 🔑
    Cloud key management — AWS KMS, Azure Key Vault, GCP KMS — keys are cloud-managed but their algorithm type, key length, and rotation policy still need to appear in CBOM for compliance.
  • 📡
    Service mesh traffic — Istio and Linkerd encrypt all pod-to-pod traffic using mTLS. The certificates for this mesh are auto-rotated but must still be tracked in CBOM for audit purposes.
🚨 Hidden Crypto Risks in Cloud
👻

Ghost Libraries

A base image like python:3.11-slim includes OpenSSL 1.1.1 (end-of-life since Sept 2023). Nobody explicitly installed it — it came with the image. Without CBOM scanning, it runs silently in production.

📅

Cert Sprawl

A Kubernetes cluster with 20 services can have 60+ certificates — service certs, ingress certs, internal CA certs, mesh certs. Manual tracking is impossible. CBOM automates the entire inventory.

🔓

Misconfigured Cloud Crypto

An S3 bucket configured with SSE-S3 (AES-128) instead of the required SSE-KMS (AES-256) is a compliance violation. CBOM surfaces this during every scan.

🐳 Container Scanning · Slide 3 of 6

CBOM in Container Images

📖 A Docker image is built in layers — each layer adding files, libraries, and configurations on top of the previous one. Cryptographic components exist across multiple layers, often introduced by the base image, not by the application developer. CBOM scanning inspects every layer to build a complete crypto inventory.

⚙️ How Container CBOM Scanning Works
  • 1️⃣
    Pull image — scanner pulls the container image from the registry (Docker Hub, ECR, GCR, ACR).
  • 2️⃣
    Extract layers — each image layer (tar archive) is extracted and all files are made accessible for analysis.
  • 3️⃣
    Scan manifests — package databases (dpkg, apk, rpm) are read to enumerate all installed packages with versions.
  • 4️⃣
    Detect crypto — crypto libraries, binaries, and certs are identified and mapped to their layer of origin.
  • 5️⃣
    Generate CBOM — structured CBOM produced with layer attribution, risk scores, and remediation guidance.
🛠️ Tools for Container CBOM

Grype — vulnerability matcher — matches CBOM components against NVD and OSS advisories
☸️ Runtime Discovery · Slide 4 of 6

CBOM in Kubernetes & Runtime Environments

📖 Container image scanning finds static crypto. But runtime discovery finds what's actually happening — which TLS versions are being negotiated between pods, which certificates are mounted in workloads, and which keys are actively in use. Kubernetes adds its own cryptographic layer on top of what's inside each container.

📡 What Runtime CBOM Discovers
  • 🔄
    Live TLS negotiation — actual cipher suites and protocol versions negotiated between services, not just what config files declare.
  • 📜
    Mounted certificates — TLS certs mounted as Kubernetes Secrets into pods, including expiry dates and issuer information.
  • 🗝️
    Active key usage — which KMS-managed keys are being called by which pods, tracked via cloud provider audit logs (CloudTrail, Cloud Audit Logs).
  • 🔁
    Cert rotation state — cert-manager and Istio auto-rotate certs; CBOM tracks rotation history and alerts if a rotation fails silently.

🎯 Real-Time Visibility: CBOM runtime scanning runs continuously as a cluster DaemonSet — every node, every pod, every service is scanned on a schedule. No manual effort, no blind spots in running workloads.
