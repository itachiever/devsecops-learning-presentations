Cryptographic Algorithms

📖 A cryptographic algorithm is a mathematical recipe — a precise set of instructions used to scramble, verify, or sign data. Algorithms are the core engine of all security. Every encryption, password check, and digital signature depends on one. CBOM tracks which algorithms are in use and whether they are still considered safe.

🗂️ Algorithm Categories Tracked in CBOM
  • 🔒
    Symmetric Encryption — same key encrypts and decrypts. Fast, used for bulk data. Example: AES-128, AES-256 (used in Wi-Fi, VPNs, disk encryption, HTTPS).
  • 🔑
    Asymmetric Encryption — key pair (public + private). Slower, used for key exchange and identity. Example: RSA-2048, RSA-4096, ECC (used in TLS handshakes, SSH).
  • 🧮
    Hashing — one-way fingerprint of data. Cannot be reversed. Example: SHA-256, SHA-3 (used in password storage, blockchain, certificate signing).
  • ✍️
    Digital Signatures — proves authenticity and integrity. Example: ECDSA, Ed25519 (used in code signing, JWT tokens, TLS certificates).

🌍 Real-World Examples
🌐

HTTPS / TLS

Uses AES-256 to encrypt your browsing session and RSA or ECDH during the handshake to securely exchange the AES key. CBOM tracks both.

🔑

Password Storage

Good apps use bcrypt or Argon2 (hashing + salting). Bad apps still use MD5 — flagged immediately in CBOM as a critical risk.

📦

Code Signing

Software packages are signed with ECDSA or RSA so your OS can verify the installer hasn't been tampered with. CBOM records the signing algorithm used.

🗝️ Component 2 of 5 · Slide 3 of 6

Cryptographic Keys

📖 A cryptographic key is a secret value — a long string of bits — that controls the output of an encryption or hashing algorithm. Same algorithm + different key = completely different output. Key management is critical: weak, short, or leaked keys make even the strongest algorithm useless. CBOM tracks every key's type, length, storage location, and rotation history.

🔵 Type 1 — Symmetric Key
🔑

One Key Does Everything

The same secret key is used to both encrypt and decrypt the data. Both sender and receiver must have an identical copy of this key — securely shared in advance.

  • Key lengths: 128-bit, 192-bit, 256-bit
  • Algorithm: AES, ChaCha20, 3DES
  • Used in: disk encryption, VPNs, HTTPS session data
  • Risk: if key is leaked → all data exposed
🟣 Type 2 — Asymmetric Key Pair
🔐

Public Key + Private Key

Two mathematically linked keys. Public key is shared with anyone — used to encrypt or verify. Private key is kept secret — used to decrypt or sign.

  • Key lengths: RSA-2048, RSA-4096, ECC-256
  • Used in: TLS certificates, SSH, JWT, digital signatures
  • Public key: safe to share — no risk in distributing
  • Private key: must NEVER leave the server
📋 What CBOM Records About Each Key
  • 📏
    Key Length — RSA-1024 is dangerously weak. CBOM flags keys below minimum safe lengths (RSA <2048, AES <128).
  • 📍
    Storage Location — Is the key in a Hardware Security Module (HSM), a secrets vault (HashiCorp Vault, AWS KMS), or hardcoded in source code? CBOM records this.
  • 🔄
    Rotation Status — Keys should be rotated regularly. CBOM tracks when a key was last rotated and alerts on stale keys.
  • 🎯
    Purpose / Scope — Is the key used for one service or shared across many? Shared keys create blast-radius risk if compromised.

⚠️ Most Common Key Risks: Hardcoded keys in source code  ·  keys never rotated  ·  RSA-1024 still active  ·  private keys stored in unencrypted config files. CBOM makes all of these visible.

📜 Component 3 of 5 · Slide 4 of 6

Certificates

📖 A digital certificate is an electronic identity document — like a passport for a website or server. It contains a public key and is signed by a trusted Certificate Authority (CA). Certificates enable trust and identity verification: your browser trusts a website only if its certificate is valid, unexpired, and signed by a recognised CA.

🗂️ What CBOM Tracks in Every Certificate
  • 🏛️
    Issuer (CA) — who signed and vouches for the certificate. Trusted CAs include DigiCert, Let's Encrypt, Comodo. An unknown issuer is a red flag.
  • 📅
    Expiry Date — certificates have a fixed lifespan (typically 90 days to 1 year). An expired cert breaks HTTPS and triggers browser "Not Secure" warnings instantly.
  • 🔑
    Public Key & Signature Algorithm — the key strength (RSA-2048, ECC-256) and the hashing algorithm used to sign it (SHA-256). Certs signed with SHA-1 are rejected by modern browsers.
  • 🌐
    Domain / Subject — which domains the cert is valid for. A certificate issued for api.example.com used on app.example.com causes errors and blocks users.
  • 🔗
    Certificate Chain — root CA → intermediate CA → leaf cert. A broken chain causes verification failures on many client devices even if the cert itself is valid.

Protocols & Cryptographic Libraries

🔗 Cryptographic Protocols

📖 A cryptographic protocol is a standardised set of rules that defines how two systems use cryptography to communicate securely — which algorithms, in which order, verified how. CBOM tracks which protocol versions are active and flags deprecated or insecure ones.

✅

TLS 1.3

Latest standard. Faster handshake, stronger cipher suites. Use for all HTTPS.

⚠️

TLS 1.2

Acceptable with correct cipher suites. Minimum standard for PCI-DSS compliance.

❌

TLS 1.0 / 1.1

Deprecated (RFC 8996). POODLE & BEAST attacks. Must be disabled immediately.

✅

SSH v2

Secure remote access standard. Strong key exchange and authentication.

❌

SSLv2 / SSLv3

Completely broken. DROWN & POODLE attacks. Remove from all servers urgently.

✅

DTLS 1.3

TLS for UDP traffic. Used in VoIP, gaming, IoT. CBOM tracks DTLS versions too.

📚 Cryptographic Libraries

📖 Crypto libraries are the software packages that implement algorithms and protocols. Applications don't write crypto from scratch — they import libraries. CBOM tracks which libraries are in use, their versions, and known vulnerabilities. A flaw in the library affects every application using it.

  • 📦
    OpenSSL — the most widely used open-source crypto library. Powers HTTPS on most of the internet. Heartbleed (CVE-2014-0160) was an OpenSSL vulnerability affecting millions of servers.
  • ☕
    BouncyCastle — popular Java/Kotlin crypto library. Used in Android apps, Spring Boot services, and enterprise Java applications.
  • 🐍
    Python Cryptography / PyCryptodome — standard crypto libraries for Python apps and data pipelines. Version matters — CBOM tracks it.
  • 🦀
    libsodium / ring — modern, memory-safe crypto libraries for Rust and C. Considered safer alternatives to legacy OpenSSL for new projects.
  • 🪟
    Windows CNG / BCryptPrimitives — Microsoft's built-in Windows crypto API. Used by .NET apps, Active Directory, and Windows services automatically.
🏁 Recap · Slide 6 of 6

Summary & Key Takeaways

⚙️

Algorithms

The math engine behind all security. CBOM flags weak ones (MD5, DES, SHA-1) and confirms strong ones (AES-256, SHA-256) are in use.

🗝️

Keys

Symmetric and asymmetric keys tracked by type, length, storage, and rotation. Hardcoded or short keys flagged as critical.

📜

Certificates

SSL/TLS cert inventory — issuer, expiry, key strength, and signing algorithm. Expired or weak certs alerted before they cause outages.

🔗

Protocols

TLS, SSH, DTLS versions tracked. Deprecated protocols (SSLv3, TLS 1.0) identified and flagged for immediate remediation.

📚

Libraries

OpenSSL, BouncyCastle, libsodium — which version is running? Library CVEs affect every app using them. CBOM tracks all.

🎯 Complete Cryptographic Visibility: A CBOM combines all 5 components into a single, unified inventory. Security teams can answer — at any moment — what algorithms run, which keys exist, which certs expire soon, which protocols are active, and which libraries carry risk. This is the foundation of cryptographic governance.
