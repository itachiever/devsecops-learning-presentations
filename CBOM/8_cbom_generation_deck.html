CBOM generation is not a manual process — it is an automated, multi-stage pipeline that discovers cryptographic components across your entire software estate, from static source code all the way to running production environments.

🔎 Step 1 of 4 · Slide 2 of 6

Step 1 — Code & Dependency Analysis

🔎 Static Analysis Stage — "Find crypto before you run the code"
⚙️ What Happens in This Stage
  • 📂
    Source Code Scanning — automated tools parse every source file (Java, Python, Go, Node.js) looking for imports, function calls, and class references to known crypto libraries and APIs (e.g., javax.crypto, openssl.*).
  • 📦
    Dependency Manifest Analysis — package lock files are parsed to enumerate all direct and transitive crypto-related dependencies with exact version strings.
  • 🔗
    Transitive Dependency Traversal — the tool recursively follows the full dependency tree. A library your code imports may itself import OpenSSL — CBOM tools capture all layers, not just the top level.
  • 🧮
    Algorithm Detection — pattern matching identifies which algorithms are being called (AES, RSA, SHA-256) and flags deprecated ones (MD5, DES, SHA-1) as immediate findings.

💡 Why Static Analysis First? It catches crypto issues before any code is deployed — the cheapest and fastest point in the software lifecycle to fix a problem. Static analysis is the foundation that all later stages build upon.

📦 Files Scanned
📄

Package Manifest Files

  • package-lock.json — Node.js/npm projects
  • pom.xml / build.gradle — Java / Maven / Gradle
  • requirements.txt / Pipfile.lock — Python
  • go.sum / go.mod — Go projects
  • Cargo.lock — Rust projects

Step 2 — Build & CI/CD Pipeline Integration

⚙️ Automated Security Checks — "Scan every build, every time"

📖 CBOM generation is embedded directly into the CI/CD pipeline — it runs automatically on every code commit, pull request, or deployment. This ensures the CBOM is always current and that crypto risks are caught before they reach production.

🚦 Policy Gate — What Gets Flagged
  • ❌
    Deprecated algorithm detected — MD5, SHA-1, DES found in any dependency → build fails with CRITICAL alert.
  • ⚠️
    Certificate expiring in < 30 days — warning issued; developers notified to initiate renewal before deployment.
  • ⚠️
    TLS 1.0 / 1.1 protocol active — flagged as high-risk; must be remediated before the next sprint's deployment.
  • ✅
    Policy compliant CBOM — build proceeds; CBOM artefact stored and signed alongside container image.

Step 3 — Runtime & Environment Discovery

🖥️ Runtime Visibility — "Find crypto that only appears when code runs"

📖 Static analysis sees what's written in code. Runtime discovery sees what's actually active — crypto libraries loaded dynamically, TLS configurations applied at startup, and certificates in use inside running containers and servers. This stage finds what static tools miss.

🔍 What Runtime Discovery Detects
  • 🐳
    Container Image Scanning — scans every layer of a Docker image for crypto libraries, certificates bundled inside, and TLS configurations baked into the image at build time.
  • 🖥️
    Running Server Inspection — tools connect to live servers and query which TLS versions and cipher suites are actually active — independently of what the config file says.
  • 🔌
    Dynamic Library Detection — some libraries are loaded only at runtime via reflection or dynamic loading. Runtime scanners detect these by observing actual process memory and loaded modules.
  • 📡
    Network Traffic Analysis — passive monitoring of TLS handshakes reveals the cipher suites and protocol versions actually negotiated between services — the ground truth of what's in use.

💡 Static vs Runtime: A config file may say TLS 1.2 only — but a runtime scan may reveal the server is still responding to TLS 1.0 handshakes due to a misconfiguration. Runtime catches this discrepancy.

Step 4 — CBOM Generation & Output

📋 Centralized Crypto Inventory — "All findings, one structured document"
🔄 How Findings Are Assembled
  • 🔗
    Merge All Sources — findings from static analysis, pipeline scans, and runtime discovery are de-duplicated and merged into a single canonical record per component.
  • 🏷️
    Enrich With Metadata — each entry is enriched with owner, location, scan timestamp, and dependency relationships before being serialised to output format.
  • 🚦
    Risk Scoring Applied — each component is scored against policy rules: CVSS scores for CVEs, algorithm strength ratings, certificate health checks. Risk level assigned: Critical / High / Medium / Low.
  • 📄
    Format & Serialise — the final CBOM is written in CycloneDX JSON or SPDX format and signed with a digital signature to prove provenance.

📦 Output Destinations: CBOM artefact stored in OCI registry alongside container image · pushed to Dependency-Track server · published to S3 / artefact repository · ingested by SIEM / security dashboard.

Summary & Key Takeaways

🔎

Step 1 — Code Analysis

Static scanning of source code and package manifests. Finds crypto imports, library versions, and algorithm calls before code is ever deployed.

⚙️

Step 2 — Pipeline Integration

CBOM scan embedded in CI/CD — every commit triggers automated crypto checks with policy gates that block weak crypto from reaching production.

🖥️

Step 3 — Runtime Discovery

Live scanning of containers, VMs, Kubernetes, and API gateways — finds crypto that only appears when the system is actually running.

📋

Step 4 — CBOM Output

All findings merged, risk-scored, and serialised to CycloneDX JSON. Signed artefact stored alongside the build for audit and compliance evidence.
