Scenario 1 — Weak Cryptography in Production

📅 How It Unfolded — The Silent Risk That Grew Over Years
✅
2020 — App Launched
MD5 chosen for password hashing.
Team thought it was "standard".
📦

⚠️
2022 — Similar Breach
Industry news: competitor

🔥

Security audit reveals MD5
still in use. 2M users at risk.
🔧
2023 — Emergency Fix
Force password reset for all users.
3-week emergency remediation sprint.

The Scenario

A fintech startup built their user authentication module in 2020 using MD5 to hash passwords — a practice that was already considered insecure but still widely copied from outdated tutorials. Over 3 years and 12 releases, this was never flagged. No developer audited the crypto layer. No automated tool scanned for deprecated algorithms. The MD5 code quietly served 2 million users.

💥

Business Impact

  • 🔴
    Security audit discovered 2M password hashes are trivially reversible using rainbow table lookups available freely online
  • 🔴
    Forced immediate password reset for all users — 23% never returned (customer churn)
  • 🔴
    Regulatory notification required under data protection laws — potential fine and mandatory public disclosure
💡

Lesson + CBOM Solution

A CBOM generated on the first build in 2020 would have flagged MD5 as NIST-deprecated immediately. A policy gate would have blocked the deployment. The team would have been forced to choose bcrypt or Argon2 on day one — before a single user was ever affected.

🎯

Key Takeaway — Weak Crypto = High Risk

Outdated algorithms don't announce themselves. They sit silently in codebases for years until exploited. CBOM makes the invisible visible — flagging deprecated algorithms at build time, not at breach time.

The Scenario

A payment services company had 34 microservices — each with its own TLS certificate managed by different teams, tracked in different spreadsheets. When the primary payment API's certificate expired at 2:47 AM, no alert fired. No monitoring existed. The on-call engineer received customer complaint calls rather than a system alert. Four hours passed before the expired certificate was identified, renewed, and traffic restored.

💥

Business Impact

  • 🔴
    4 hours of complete payment outage — estimated revenue loss of $180,000 for a mid-sized operator
  • 🔴
    Browser "Not Secure" warning triggered trust erosion — 12% of users never returned post-incident
  • 🔴
    Regulatory filing required as a critical service availability incident under financial sector guidelines
💡

Lesson + CBOM Solution

CBOM automatically inventories all 34 service certificates and tracks each expiry date. Alerts are sent 90, 30, and 7 days before expiry — to named owners. cert-manager integration auto-renews 95% of certs before any human action is required. The 2:47 AM call never happens.

🎯

Key Takeaway — Certificates Must Be Actively Managed

A certificate expiry is 100% predictable and 100% preventable. CBOM converts a surprise outage into a managed, scheduled renewal event — with owners, SLAs, and automation taking the risk to near zero.

🐛 Case Study 3 · Slide 4 of 6

Scenario 3 — The Vulnerable Crypto Library Nobody Knew About

🚨

The Pattern: A known-vulnerable version of a cryptographic library silently runs inside a container image that no developer on the team explicitly installed — introduced by a base image three layers deep in the Docker build hierarchy.

📖

The Scenario

A DevOps team deployed their API gateway using node:16-slim as the base Docker image. Unknown to them, this image included OpenSSL 1.1.1q — a version with CVE-2022-3786 (CVSS 7.5, High). Three months after deployment, a security scanner run as part of an insurance audit found it. By then, the vulnerable endpoint had processed 18 months of sensitive API traffic.


The team had never considered scanning their container image. They believed they were responsible only for the code they wrote. They were not aware they were shipping OpenSSL at all — let alone a vulnerable version of it.

💥

Business Impact

  • 🔴
    18 months of exposure to a High-severity CVE in production — all traffic potentially at risk of TLS-level attack during this window
  • 🔴
    Emergency container rebuild required for all 12 services using the same base image — unplanned 2-week security sprint
  • 🔴
    Insurance audit flagged as a material security control failure — increased premiums and mandatory quarterly image scanning requirement
💡

Lesson + CBOM Solution

A CBOM generated from the container image at build time would have listed OpenSSL 1.1.1q with its version and PURL. CVE matching against NVD would have returned CVE-2022-3786 immediately. The CI/CD gate would have blocked deployment. The team would have upgraded to OpenSSL 3.0.7 before the first API request was ever served.

🎯

Key Takeaway — Hidden Dependencies Carry Real Risk

You don't choose every component your application ships. Your base image chooses many for you. CBOM scans every layer of the container image — making every transitive, inherited crypto dependency visible, evaluated, and governed.
