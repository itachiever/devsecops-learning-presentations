Basic Interview Questions — Fundamentals Matter Most

📌 Interviewer tip: Basic questions test conceptual clarity. Avoid over-complicating answers — a concise, accurate definition with one real-world example is more impressive than a long rambling response. Lead with the definition, follow with the purpose, close with an example.

Q
What is a CBOM (Cryptographic Bill of Materials)?Basic
A
A CBOM is a structured inventory of all cryptographic components in a software system — algorithms, keys, certificates, and protocols — captured in a machine-readable format like CycloneDX JSON. It is the "ingredient list" for the security layer of your software. CycloneDXAlgorithmsCertificates
Q
Why is CBOM needed in modern software?Basic
A
Because most organizations don't know what cryptographic components are running in their applications — especially transitive dependencies and container base images. CBOM provides visibility that enables vulnerability detection, certificate lifecycle management, compliance reporting, and post-quantum migration planning. Without it, crypto risks are invisible. VisibilityRiskCompliance
Q
What is the difference between SBOM and CBOM?Basic
A
SBOM (Software Bill of Materials) inventories all software components — libraries, packages, versions. CBOM is a specialized subset that focuses specifically on cryptographic components — algorithms, keys, certs, protocols. CBOM extends SBOM by adding crypto-specific fields that standard SBOM tools don't capture. Think of SBOM as the full ingredient list and CBOM as the security-specific nutrition facts. SBOMSubsetCycloneDX 1.5+

🟡 Intermediate Level · Slide 3 of 6

Intermediate Questions — Practical Understanding

Q
What are the main components tracked in a CBOM?Intermediate
A
A CBOM tracks four core component types: (1) Algorithms — encryption, hashing, signing (AES-256, SHA-256, RSA); (2) Keys — type, bit-length, storage location, rotation policy; (3) Certificates — issuer, expiry, signature algorithm, revocation status; (4) Protocols — TLS version, SSH version, cipher suites in use. Each entry includes the component name, version, PURL, owner, and usage location.
Q
How is a CBOM generated? What tools are used?Intermediate
A
CBOM generation happens at three levels: (1) Static — tools like cdxgen and Syft scan source code and package manifests at build time; (2) Container — Trivy scans every Docker image layer; (3) Runtime — Kubernetes DaemonSet scanners probe live pods and cloud APIs. Output is a signed CycloneDX 1.5+ JSON artefact stored alongside the container image.
Q
Where does CBOM fit in a CI/CD pipeline?Intermediate
A
CBOM integrates at multiple CI/CD stages: (1) Build stage — CBOM generated after compilation, before any deployment; (2) Scan stage — CBOM enriched with CVE matches from Dependency-Track; (3) Policy gate — build fails if CBOM contains Critical findings (deprecated algorithm, expired cert); (4) Artefact store — signed CBOM pushed to registry alongside the image; (5) Runtime monitoring — scheduled re-scan refreshes CBOM continuously in production. 🔴 Advanced Level · Slide 4 of 6

Advanced Questions — Real-World Application

Q
How does CBOM help in vulnerability management?Advanced
A
CBOM enables precise, automated CVE mapping. Each CBOM component entry contains a PURL (Package URL) — e.g., pkg:generic/openssl@3.0.5. Tools like Dependency-Track cross-reference every PURL against NVD, GitHub Advisory, and OSS Index in real time. When a new CVE drops — like CVE-2022-3602 for OpenSSL — all CBOM entries with that version are flagged instantly, owners are notified, and the severity score (CVSS 9.8) drives remediation priority. Without CBOM, the same discovery requires manual investigation that can take weeks. CVE MappingNVDCVSS ScoringDependency-Track
Q
How does CBOM detect weak cryptography?Advanced
A
CBOM stores the exact algorithm name, mode, and key size for every crypto component. A policy-as-code file (versioned in Git) defines the approved-algorithm list — e.g., AES-256-GCM approved, AES/ECB flagged, MD5 blocked, TLS 1.0 blocked. On every CBOM scan, each component is evaluated against this policy. Violations are classified by severity: MD5 or DES → Critical; SHA-1 → High; TLS 1.2 with weak cipher suites → Medium. The CI/CD gate blocks deployment on Critical findings. Policy-as-CodeMD5 / SHA-1TLS 1.0Build Gate
Q
How does CBOM support CERT-In compliance?Advanced
A
CERT-In mandates that organizations maintain a complete, structured, continuously updated cryptographic inventory. CBOM directly satisfies this requirement by capturing all four mandatory field groups: component identification, cryptographic details, certificate metadata, and usage/dependency information — in a signed CycloneDX 1.5+ document. The timestamp field proves currency. The digital signature proves authenticity. The PURL enables automated CVE cross-referencing. When a CERT-In auditor requests evidence of cryptographic visibility, the CBOM — generated at every build — provides timestamped, signed, machine-readable proof on demand. Scenario-Based Questions — Problem-Solving Mindset

📌 Interviewer tip: Scenario questions test your structured thinking under pressure. Always state your approach before diving into steps: "I'd first assess scope, then prioritise by impact, then remediate, then verify." Interviewers reward candidates who think systematically, not just technically.

Scenario A: Expired Certificate Discovered in Production at 2 AM

Scenario

📍 The Situation

The payment API gateway shows "ERR_CERT_DATE_INVALID". Customers are getting browser security warnings. The on-call engineer receives alerts at 2 AM. No one knows which certificate expired or who owns it. How do you respond — and how would CBOM have prevented this?

✅ Your Structured Response

1
Identify: Query CBOM platform for all certificates with expired/expiring status — find the payment API cert within 30 seconds
2
Escalate: CBOM owner field shows the responsible team — direct contact in under 2 minutes, not a 30-minute search
3
Renew: Emergency cert renewal or activate the pre-staged replacement cert. Rollout via CI/CD for traceability
4
Prevention: CBOM with 90/30/7-day alerts would have triggered automated renewal via cert-manager — this incident should never have happened
💀

Scenario B: Security Scan Reveals MD5 Used for Token Signing in Auth Service

Scenario

📍 The Situation

A CBOM scan flags a Critical finding: the auth service is using MD5 to sign session tokens. MD5 is cryptographically broken. The service handles 500,000 daily active users. How do you handle this — from discovery to remediation?

✅ Your Structured Response

1
Assess impact: CBOM dependency map shows which services consume auth tokens — understand blast radius before acting
2
Prioritise as Critical: MD5 CVSS equivalent = Critical. 24-hour SLA. No exceptions without CISO-signed risk acceptance
3
Migrate: Replace MD5 with SHA-256 or HMAC-SHA256. Plan token invalidation strategy for existing sessions
4
Verify: Re-scan CBOM post-fix — confirm SHA-256 now appears. CBOM gate blocks re-introduction of MD5 in future builds
