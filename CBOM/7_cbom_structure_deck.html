A CBOM is not a free-form document — it follows a strict, standardised structure where every cryptographic component is represented as organised, nested data. Understanding this structure is the key to reading, generating, and automating CBOM workflows in DevSecOps.

📌 Standard: The primary CBOM format is defined by CycloneDX 1.5+ (OWASP) — an open, machine-readable specification that extends SBOM with a dedicated cryptographic-asset component type.

🗂️ Foundation · Slide 2 of 6

What is CBOM Structure?

📖 CBOM is structured data — not plain text, not a spreadsheet, not a PDF. Every piece of information lives in a defined field, inside a defined section, with defined relationships. This makes it readable by both humans and automated security tools equally well.

📋 CBOM Document Top-level container — one document per application / service
📋 Header & Metadata
Format · Version · Serial No. · Timestamp · Tool
📦 Components [ ]
Array of crypto-asset entries — one per component
🔗 Dependencies [ ]
Maps which services use which components
🔎 Vulnerabilities [ ]
CVE / risk entries linked to components

🏗️ Structured means: Every CBOM tool — whether it's cdxgen, Syft, or a custom scanner — produces output in the same predictable shape. A security dashboard can ingest any valid CBOM without custom parsing.

🤖 Machine-readable means: CI/CD policy engines, SIEMs, and vulnerability trackers can automatically parse every field — triggering alerts, blocking builds, and generating compliance reports with zero human intervention.

CBOM Data Organization — Logical Field Groups

📖 Inside the Components array, each crypto-asset entry is divided into four logical field groups. Data is never mixed randomly — it is precisely grouped so any reader immediately knows where to look for what type of information.

📄 Formats · Slide 4 of 6

CBOM Format Types — How the Data is Written

📖 A CBOM document is serialised (written out) in a machine-readable format — a standardised text notation that both humans and tools can parse. The format choice affects which tools can read the CBOM, how it integrates into pipelines, and how it is stored.

{ }

JSON — JavaScript Object Notation

✅ Primary Format
StandardCycloneDX 1.5+ JSON Schema
ReadabilityHuman-readable with basic formatting
ToolingNative support in all DevSecOps tools
File sizeCompact — ideal for CI/CD artefacts
Use caseREST APIs, artefact registries, SIEM ingest
Extension.cdx.json
</>

XML — Extensible Markup Language

⚠️ Legacy / Enterprise
StandardCycloneDX XML Schema (same spec)
ReadabilityVerbose — more tags than data
ToolingPreferred by legacy ITSM / enterprise systems
File sizeLarger — due to tag overhead
Use caseSAP integrations, legacy audit systems
Extension.cdx.xml

🌐 SPDX Format — an alternative open standard from the Linux Foundation. Primarily designed for SBOM but supports cryptographic fields. Used widely in open-source and government contexts. File extension: .spdx.json

🔏 Digital Signature: A valid CBOM is cryptographically signed using JSON Web Signatures (JWS) — this proves the CBOM was generated by a specific trusted tool and has not been tampered with since generation.

🔗 Key Structural Concepts
  • 🪆
    Nesting — crypto properties are nested inside the component entry, not listed separately. Each component owns its own algorithm, key, and certificate data.
  • 🔗
    Relationships — the dependencies section separately maps which application services reference which CBOM component by its unique ID — creating a traceable graph.
  • 🎯
    Risk at Component Level — each component carries its own risk score, computed from algorithm strength + cert health + key rotation status. One field, instant overview.
🚦 Risk Indicators in Structure

✅ riskLevel: "low" — AES-256 + TLS 1.3 + cert valid 365d + key in HSM. All fields compliant with FIPS and CERT-In guidelines.

⚠️ riskLevel: "medium" — TLS 1.2 (acceptable) + cert expires in 14 days. Action required: schedule cert renewal this sprint.

❌ riskLevel: "critical" — algorithm: MD5 detected. Flagged for immediate removal. Build gate blocks deployment until resolved.

🏁 Full Recap · Slide 6 of 6

Summary & Key Takeaways

🏗️

Structured

CBOM is not free-form text. Every field lives in a defined position — header, components array, dependencies, and vulnerabilities — in a strict, predictable hierarchy.

🗂️

Organized

Data is grouped into 4 logical sections per component: identification, crypto properties, certificate metadata, and usage/dependency info. Nothing is mixed together.

🤖

Machine-Readable

Serialised in JSON (primary) or XML — formats that CI/CD pipelines, SIEMs, and policy engines can parse, query, and act on automatically without any human intervention.

🎯 How Structure Enables Security Outcomes
⚙️

Automation

Predictable structure means tools can generate, parse, and compare CBOMs automatically — no manual data entry, no human error in the critical security data layer.

🔌

Integration

Standard formats (CycloneDX JSON) plug directly into Dependency-Track, GitHub Advanced Security, Snyk, OWASP tools, and custom dashboards without conversion.

🛡️

Security Management

Nested risk fields and dependency maps let security teams instantly see blast radius, prioritise by risk level, and prove compliance to auditors with one structured document.
