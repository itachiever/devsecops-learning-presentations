What is SBOM?

❓ Key Question: "What software components are being used?"
📖 Definition

A Software Bill of Materials (SBOM) is a complete, machine-readable inventory of all software components inside an application — like a detailed ingredient list for your code. It tells you exactly what is inside, without examining each component individually.

📦 What Does an SBOM Track?
  • 📚
    Libraries & Packages — every open-source and commercial library used (e.g., React, Spring Boot, OpenSSL, log4j).
  • 🔗
    Direct & Transitive Dependencies — not just what you imported, but every package those packages depend on — potentially hundreds deep.
  • 🏷️
    Versions & Licenses — exact version numbers and open-source licenses (MIT, GPL, Apache) to track legal obligations.
  • 🏛️
    Supplier & Origin — who created and maintains each component, and where it was sourced from.
🌍 Real-World Example
🛒

E-Commerce Application SBOM

Your app uses React, Axios, Express, and lodash directly. But lodash pulls in 4 more packages, Express pulls in 12, and Axios pulls in 3. Your SBOM captures all 20+ components — giving you a complete picture.

✅ Why It Matters

🛡️ Supply chain security — Executive Order 14028 (US) mandates SBOMs for federal software. Knowing your components is now a compliance requirement.

⚡ Rapid incident response — When Log4Shell dropped in 2021, organizations with SBOMs found affected systems in hours. Others took weeks.

📋 SBOM Formats: CycloneDX (OWASP) and SPDX (Linux Foundation) are the two dominant open standards. Tools like Syft, cdxgen, and Trivy can auto-generate SBOMs from any codebase.

🔍 Concept 2 of 3 · Slide 3 of 6

What is SCA?

❓ Key Question: "Are the components I use vulnerable?"
📖 Definition

Software Composition Analysis (SCA) is the process of automatically scanning your dependencies for known security vulnerabilities. SCA takes the inventory from an SBOM and cross-references every component against vulnerability databases like the National Vulnerability Database (NVD) and CVE records.

⚙️ How SCA Works (Step by Step)
  • 1️⃣
    Scan — SCA tool scans your codebase, package files (package.json, pom.xml, requirements.txt), and containers.
  • 2️⃣
    Identify — Builds a component list (often using SBOM internally) with exact versions of every library.
  • 3️⃣
    Match — Cross-references component versions against CVE databases, GitHub Advisory, Snyk DB, and OSS Index.
  • 4️⃣
    Report — Outputs a prioritised list of vulnerabilities with severity scores (CVSS) and recommended fix versions.
🌍 Real-World Example
🚨

Detecting a Vulnerable Package

Your app uses log4j 2.14.1. SCA instantly flags CVE-2021-44228 (Log4Shell — CVSS 10.0 Critical) and tells you the fix is log4j 2.17.1. Without SCA, you'd never know until an attack happened.

🛠️ Popular SCA Tools
  • 🔍 Snyk — developer-first, IDE integration, real-time alerts
  • 🔍 OWASP Dependency-Check — open-source, CI/CD pipeline ready
  • 🔍 Trivy — fast scanner for containers, filesystems, and git repos
  • 🔍 Dependabot — built into GitHub, automatic PR updates

⚡ SCA vs SBOM: SBOM answers "what is here?" — SCA answers "is it safe?". SCA uses SBOM data as its starting point. They are complementary, not competing tools.

🔐 Concept 3 of 3 · Slide 4 of 6

What is CBOM?

❓ Key Question: "How is my data cryptographically secured?"
📖 Definition

A Cryptographic Bill of Materials (CBOM) is a structured inventory of all cryptographic components in a software system — including which algorithms protect your data, what keys are in use, which certificates are active, and which protocols carry your traffic.

🗂️ What Does a CBOM Track?
  • ⚙️
    Algorithms — AES-256 (symmetric), RSA-2048 (asymmetric), SHA-256 (hashing). Flags deprecated ones like MD5 and DES.
  • 🗝️
    Cryptographic Keys — type, length, storage location, and rotation status. Weak or short keys are highlighted.
  • 📜
    Certificates — TLS/SSL certificates, issuer, expiry date, key strength. Alerts on expiring or mis-configured certs.
  • 🔗
    Protocols — TLS 1.2/1.3, SSH, DTLS. Deprecated protocols like TLS 1.0/SSLv3 are flagged as critical risk.
🌍 Real-World Example
🏦

Banking Application CBOM

A CBOM scan reveals: payment service uses AES-256 ✅, auth service uses MD5 for token signing ❌, API gateway runs TLS 1.0 ❌. Without CBOM, these risks were completely invisible to the team.

🆚 Why CBOM is Different from SBOM + SCA

📦 SBOM tells you log4j 2.14.1 is present. SCA tells you it has a CVE. But CBOM tells you how data is encrypted inside — a completely different security dimension SBOMs and SCA tools do not cover.

⚛️ Only CBOM enables post-quantum cryptography (PQC) readiness — identifying which algorithms will break under quantum computing and mapping migration paths.

🛠️ CBOM Tools: cdxgen (CycloneDX CBOM extension) · IBM SBOM Utility · Cryptosense Analyzer · custom CI/CD crypto scanners in DevSecOps pipelines

📊 Full Comparison · Slide 5 of 6

SBOM vs SCA vs CBOM — Side-by-Side Comparison

Tool Purpose What It Tracks Output / Result Primary Use Case
📦 SBOM Software inventory — catalogues every component inside an application Libraries, packages, dependencies, versions, licenses, suppliers Structured component list (CycloneDX / SPDX format) — machine & human readable Supply chain transparency · compliance (EO 14028) · dependency management · incident response
🔍 SCA Vulnerability detection — finds known security flaws in your dependencies CVEs, license violations, outdated versions, malicious packages Prioritised vulnerability report with CVSS severity scores and remediation versions Shift-left security · CI/CD pipeline scanning · patch management · risk prioritisation
🔐 CBOM Cryptographic visibility — inventories all crypto components and their health Algorithms, keys, TLS certificates, protocols, cipher suites, key lengths Crypto inventory with risk flags — weak algorithms, expired certs, insecure protocols Crypto governance · PQC readiness · cert lifecycle · compliance (FIPS, PCI-DSS, HIPAA)
📦

"What is here?"

SBOM maps the complete ingredient list of your software — every component, every version.

🔍

"Is it vulnerable?"

SCA checks each component against known CVE databases and flags exploitable weaknesses.

🔐

"How is data secured?"

CBOM audits the cryptographic layer — ensuring algorithms, keys, and certs are strong and current.

💡 Key Insight: SBOM, SCA, and CBOM are not competing — they are complementary layers. SBOM provides the foundation. SCA analyses it for vulnerabilities. CBOM goes deeper into the cryptographic security posture.
