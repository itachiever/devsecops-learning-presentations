What are Digital Certificates?

📖 A digital certificate is an electronic identity document — like a passport for a website or server. It contains a public key, the owner's identity, validity dates, and a digital signature from a trusted Certificate Authority (CA) that vouches for its authenticity. Without it, browsers cannot verify a site's identity.

Common Certificate Issues & Real-World Impact

📖 Certificate management failures are one of the most common causes of production outages and security incidents. Unlike application bugs that are caught in testing, certificate issues often appear silently — until the moment the cert expires, causing an immediate, total service failure visible to every user.

⏰

Expired Certificate

🔴 Critical
Certificate passed its Not After date. Browsers immediately show "Your connection is not private" (ERR_CERT_DATE_INVALID). All HTTPS traffic is blocked. Affects 100% of users instantly.
💥 Impact: Complete service outage · users blocked · revenue loss · brand damage
⚙️

Misconfigured Certificate

🟠 High
Common misconfigurations: cert issued for api.example.com installed on app.example.com (domain mismatch). Broken certificate chain — intermediate CA not installed. Self-signed cert in production.
💥 Impact: Browser warnings · mobile app failures · API clients throw SSL errors
🔍

Unknown Certificate Usage

🟡 Medium
Shadow IT and automated deployments create certificates that nobody tracks. A microservice spun up 18 months ago has a cert nobody knows about — and nobody will know when it expires. Discovered only in outage.
💥 Impact: Unexpected outages · no owner to contact · slow incident response
🔐

Weak Signature Algorithm

🔴 Critical
Certificates signed with SHA-1 are rejected by Chrome, Firefox, and Safari since 2017. Even if the cert is valid and not expired, browsers will refuse to trust it — causing the same outage as an expired cert.
💥 Impact: Distrust by modern browsers · TLS handshake failure · compliance violation
🚫

Revoked Certificate

🔴 Critical
CA revokes a cert if the private key was compromised or the domain owner requested it. Browsers check OCSP / CRL on every connection. A revoked cert in production causes immediate distrust errors.
💥 Impact: Immediate HTTPS failure · potential ongoing security breach if key was stolen
📈

Certificate Sprawl

🟠 High
Enterprises with 50+ microservices, multiple cloud accounts, and different teams can accumulate hundreds of certificates across environments. Manual tracking in spreadsheets always falls behind reality.
💥 Impact: Missed renewals · no single source of truth · audit failures · expiry surprises

Certificate Management Best Practices with CBOM

📖 Effective certificate management is proactive, automated, and continuous. The goal is to never discover a certificate problem reactively — through an outage or a user complaint. CBOM provides the inventory layer that makes all the following best practices enforceable and auditable.

📅

Monitor Expiry Dates Continuously

1
CBOM scans all environments daily and updates expiry countdowns in real time for every certificate
2
Alert thresholds configured: 90-day, 30-day, 14-day, and 7-day notifications to certificate owners
3
Dashboard shows traffic-light status (green/amber/red) — zero spreadsheets, zero manual checks needed
🤖

Automate Certificate Renewal

1
Deploy cert-manager in Kubernetes — auto-renews Let's Encrypt certificates 30 days before expiry
2
Use AWS ACM / Azure Key Vault for cloud services — fully managed renewal with zero manual steps
3
CBOM verifies renewal succeeded after each run — alerts if a scheduled renewal failed silently
🔄

Rotate Certificates Regularly

1
Set maximum cert lifetime to 90 days (Let's Encrypt default) — reduces blast radius if a key is compromised
2
Internal mTLS certs (Istio, Linkerd) rotate every 24 hours automatically — CBOM validates rotation health
3
After any security incident involving a private key, CBOM flags all certs using that key for immediate revocation
📋

Maintain Complete Inventory

1
Every new service deployment automatically adds its cert to CBOM — no manual registration required
2
CBOM tags each cert with: owner, environment (prod/staging), service name, and last-verified timestamp
3
Decommissioned services trigger cert revocation review — CBOM prevents orphaned certs from lingering
