Execution Layer
What is a Worker Node?
🖥
A machine that runs your applications — can be a physical server or a cloud virtual machine
📋
Executes workloads assigned by Control Plane — it never decides on its own, just follows instructions
📦
Hosts Pods — a Pod is the smallest unit in K8s, wrapping one or more containers (your apps)
📡
Reports back to Control Plane — constantly sends health status and metrics to keep K8s updated
Analogy: Worker Node = a factory floor 🏭 — the Control Plane is management, Worker Nodes do the actual production

Node Internals
Worker Node — Component Overview
🤝
Kubelet
Node Agent
The supervisor on every node. Receives Pod specs from the API Server and makes sure containers are running correctly.
✅ Runs on every worker node
✅ Monitors container health
✅ Reports status to API Server
= Supervisor / Node Agent
🐳
Container Runtime
containerd / CRI-O
The engine that actually pulls images and starts containers. Works under Kubelet's direction via the CRI interface.
✅ Pulls container images
✅ Starts & stops containers
✅ Works via CRI interface
= Container Engine
🌐
Kube-proxy
Network Proxy
The network manager of each node. Maintains routing rules so traffic reaches the right Pod from any Service.
✅ Pod-to-Pod communication
✅ Service-based routing
✅ Uses iptables / IPVS
= Network Router
Together: Kubelet manages → Runtime runs → Kube-proxy connects. All three must work together to run your applications correctly.

Component 1
Kubelet — The Node Agent
📍
Runs on every worker node — it's always present, always listening for instructions
📥
Receives Pod specs from API Server — a PodSpec (YAML) tells it exactly what containers to run
🔍
Monitors container health — uses liveness & readiness probes to check if containers are okay
📤
Reports status back to API Server — "Pod is running, healthy, using X CPU/memory"
🔁
Restarts failed containers — if a container crashes, Kubelet tells runtime to restart it

Container Runtime — The Engine
⚙️
Responsible for running containers — the actual engine underneath Kubernetes
📥

🚀
Starts and stops containers — creates isolated environments using Linux namespaces & cgroups
🔗
Works with Kubelet via CRI — CRI = Container Runtime Interface, the standard API between them

Component 3
Kube-proxy — The Network Manager
🌐
Handles all networking on the node — manages how traffic flows in, out, and between Pods
🔗
Enables Pod-to-Pod communication — Pods on different nodes can still talk to each other
⚖️
Service-based routing — routes traffic from a Service to the correct backend Pods
📋
Maintains network rules — uses iptables or IPVS to set routing rules on every node
Simple: kube-proxy = the node's traffic cop 🚦 — it directs every network packet to the right destination
