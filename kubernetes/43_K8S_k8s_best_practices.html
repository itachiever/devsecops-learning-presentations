Foundation
Why Best Practices Matter
💥
Poor configuration directly causes outages — a container with no memory limit hogs a node, starving other Pods. No liveness probe means a deadlocked app keeps running. Small config mistakes = production incidents.
🔓
Misconfiguration is the #1 Kubernetes security risk — running containers as root, overly permissive RBAC, exposed secrets in env vars. Most K8s breaches aren't zero-days — they're configuration mistakes.
📈
Production workloads demand reliability — a cluster serving millions of requests needs liveness probes, resource limits, multi-AZ scheduling, and rolling deployments. None of these are optional in prod.
🏆
Best practices are interview essentials — "How do you secure a K8s cluster?" and "What's the difference between liveness and readiness probes?" are top DevOps interview questions. Know these cold.
Analogy: K8s best practices = building codes for a skyscraper 🏙. You can build without them, but one structural oversight can bring the whole thing down. Follow the codes.

Practice #1
Always Set Resource Requests & Limits
📊
Requests = guaranteed resources for the scheduler — Kubernetes uses requests to decide which node has enough capacity for a Pod. Without requests, the scheduler can't make good placement decisions.
🚧
Limits = maximum — protects other Pods — without limits, one runaway container can consume all node RAM, causing OOMKills of innocent neighboring Pods. Limits are your blast radius control.
⚖
Set limits 2–3× above average usage — too tight = constant OOMKills. Too loose = poor bin packing. Profile your app under load, then set limits at 2× p95 memory usage.
⚠
No limits = Burstable QoS class — Kubernetes has 3 QoS classes: Guaranteed (req=limit), Burstable (partial), BestEffort (none). BestEffort Pods are evicted first under pressure. Don't be BestEffort in prod.
Correct YAML Example
resources:
  requests: # Scheduler uses this
    cpu: "250m" # 0.25 core
    memory: "256Mi"
  limits: # Max allowed
    cpu: "500m"
    memory: "512Mi"
✅ Guaranteed QoS
request = limit → highest priority · never evicted first
⚠ Burstable QoS
request < limit → ok for dev, risky for prod
❌ BestEffort QoS
no requests or limits → evicted FIRST under pressure

Practice #2
Secure Kubernetes Workloads
🔐
RBAC
Least privilege access
Grant only the permissions each service account needs. A web app shouldn't have cluster-admin. Use separate ServiceAccounts per application. Audit RBAC regularly.
verbs: ["get","list"] — not ["*"]
🚫
Non-root Containers
Never run as root
If a container running as root is compromised, the attacker has root on the node. Set runAsNonRoot: true and runAsUser: 1000 in securityContext.
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  readOnlyRootFilesystem: true
🔑
Secrets Management
Never in env vars or YAML
Never put passwords in Deployment YAML or env vars committed to Git. Use K8s Secrets (base64 ≠ encryption!), and ideally Vault, Sealed Secrets, or cloud KMS for true encryption.
Base64 is NOT encryption ⚠
🌐
Network Policies
Default deny all
By default, all Pods can talk to all other Pods. Use NetworkPolicies to implement a deny-all baseline, then explicitly allow only required traffic flows between services.
Micro-segmentation = zero-trust
🔍
Image Scanning
Block vulnerable images
Scan images in CI with Trivy, Snyk, or Grype before pushing to registry. Use Binary Authorization or ImagePolicyWebhook to block unscanned or high-severity images in production.
trivy image myapp:v1.2 ✅

Practice #3
Always Configure Health Probes
❤
Liveness Probe — "Is this container alive?" — Checks if the app is still running. If the probe fails, Kubernetes restarts the container. Catches deadlocks, infinite loops, and zombie processes.
✅
Readiness Probe — "Is this container ready for traffic?" — Checks if the app can serve requests. If the probe fails, Kubernetes removes it from the load balancer without restarting. Catches slow startup and overload.
🚀
Startup Probe — "Has the app finished starting?" — For apps with slow initialization. Delays liveness checks until startup completes. Prevents restart loops during initial boot of Java/Spring apps.
⚠
Never skip probes in production — without probes, Kubernetes sends traffic to containers that crashed or are deadlocked. Users get errors until a human notices. Probes = automatic self-healing.
Rule of thumb: Liveness = "should I restart this?" · Readiness = "should I send traffic to this?" · Startup = "has this finished booting?" All three serve different purposes.
Probe YAML
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  failureThreshold: 3
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 10
❤ Liveness fails → restart container
✅ Readiness fails → remove from LB
🚀 Startup probe → delay liveness

Practice #4
Monitor & Log Everything
📊
Metrics — the pulse of your cluster — Prometheus scrapes CPU, memory, request rates, error rates, and latency every 15 seconds. Grafana dashboards make these visible. You cannot optimize what you don't measure.
📋
Centralized logging is non-negotiable — Pods die, logs disappear. Deploy Fluent Bit as a DaemonSet to ship all logs to Elasticsearch/Loki before Pods terminate. Centralized = searchable across all Pods.
🔔
Alerts must be actionable — alert on the Four Golden Signals: Latency, Traffic, Errors, Saturation. Every alert should require human action. Alert fatigue kills on-call culture. Quality over quantity.
🔗
Correlate metrics + logs + traces — when an alert fires, link from Grafana directly to the relevant Kibana/Loki logs. Implement distributed tracing (Jaeger, Zipkin) for microservice debugging.
Stack: Prometheus + Grafana for metrics & alerts. Fluent Bit + Loki/EFK for logs. Jaeger/Tempo for distributed tracing. The three pillars of observability.
Observability Stack
📊 Metrics
Prometheus scrapes → Grafana visualizes → Alertmanager notifies
📋 Logs
Fluent Bit (DaemonSet) → Loki / Elasticsearch → Kibana / Grafana
🔗 Traces
OpenTelemetry → Jaeger / Tempo → end-to-end request tracing
🔔 Alert Rules (example)
CPU > 80% for 5m → page
Error rate > 1% → Slack
Pod restarts > 3 → warn

Practice #5
Use CI/CD & Automation
🤖
Automate everything — zero manual kubectl in prod — manual kubectl apply is the enemy of reliability. Every production change should go through a CI/CD pipeline with tests, review, and automated rollback capability.
🌿
GitOps = Git as the single source of truth — all Kubernetes manifests live in Git. Argo CD or Flux watches the repo and auto-syncs the cluster. Every change is auditable, reviewable, and reversible.
📋
Infrastructure as Code — Helm or Kustomize — never manage clusters by hand. Use Helm Charts or Kustomize overlays so your entire infrastructure is version-controlled, reproducible, and testable.
🧪
Test before every deployment — unit tests, integration tests, image scanning, policy checks (OPA/Kyverno). A deployment pipeline that skips tests is not a pipeline — it's a gamble.
Automation goal: From git push to production without a human touching a keyboard. The pipeline builds, tests, scans, deploys, and validates. Humans set policy — machines execute it.
GitOps Pipeline
💻 git push → PR → Review
↓
🔨 Build + Test + Scan
↓
🐳 Push Image (tagged SHA)
↓
🐙 Argo CD detects change
↓
✅ K8s rolling deploy
Auto rollback on failure ⏪

Practice #6
Use Namespaces & Labels Properly
📁
Namespaces = logical isolation boundaries — separate teams, environments, and applications. Namespaces enable RBAC per team, resource quotas per env, and NetworkPolicies between groups. Default namespace = bad practice in prod.
🔒
Set ResourceQuotas per namespace — prevent one team from consuming all cluster resources. Define max CPU, memory, Pod count, and PVC count per namespace. Enforced automatically by the scheduler.
🏷
Label everything consistently — use standard labels: app, env, team, version. Labels enable cost attribution, monitoring queries, NetworkPolicies, and Service selectors.
🏗
Recommended namespace structure — kube-system (K8s components), monitoring (Prometheus/Grafana), production, staging, one per team.

Practice #7
Backup & Disaster Recovery
💾
Back up etcd — your cluster's brain — etcd stores all cluster state. If it's lost without a backup, your entire cluster configuration, deployments, and resources are gone permanently. Snapshot etcd daily at minimum.
🗄
Use Velero for full cluster backups — Velero backs up Kubernetes resources (YAML manifests) and Persistent Volumes to cloud object storage (S3, GCS, Azure Blob). Restores entire namespaces or clusters.
📋
Persistent Volume backups are critical — stateful apps (databases, file storage) write to PVs. Back up PVs using Velero with restic/kopia, cloud snapshots, or application-native tools (pg_dump, mysqldump).
🧪
Test restoration regularly — not just backups — an untested backup is not a backup. Schedule quarterly disaster recovery drills. Restore to a test cluster. Validate data integrity. Document the RTO and RPO achieved.
Key terms: RPO (Recovery Point Objective) = how much data you can lose. RTO (Recovery Time Objective) = how long to recover. Both must be defined before a disaster, not during.
Backup Strategy
⎈ etcd Backup
Daily snapshot → S3/GCS. Retain 30 days. Test restore monthly.
📦 Velero Cluster Backup
All K8s resources + PV snapshots to object storage. Schedule: */6h.
🗄 Database Backup
pg_dump/mysqldump to S3 daily. Point-in-time recovery (PITR) enabled.
🧪 DR Test Schedule
Quarterly: restore full cluster to staging · measure RTO/RPO · document findings
velero backup create my-backup --include-namespaces production

Practice #8
Real-World Production Practices
☁
Managed K8s
Use EKS, GKE, or AKS instead of self-managed. Cloud providers handle control plane HA, upgrades, and patches. Focus engineering time on apps, not infra ops.
Self-managed only when necessary
⬆
Regular Upgrades
K8s N-2 support policy — only last 3 versions receive security patches. Clusters 2+ versions behind become security liabilities. Upgrade every 3–4 months using rolling node group upgrades.
Stay within N-2 versions always
🔍
Image Scanning
Scan every image in CI/CD before deploy. Use Trivy, Grype, or Snyk. Block images with CRITICAL/HIGH CVEs. Re-scan images in registry weekly as new CVEs are disclosed.
trivy image --exit-code 1 app:latest
📈
Smart Autoscaling
Configure HPA (Pod scaling), VPA (right-sizing), and Cluster Autoscaler (node scaling). Use Spot/Preemptible instances for non-critical workloads to cut costs by 60–80%.
Spot nodes = 70% cost reduction
📖
Documentation
Document runbooks for every alert, architecture decisions, onboarding guides, and incident postmortems. A cluster only your senior engineer understands is a single point of failure.
Every alert needs a runbook
