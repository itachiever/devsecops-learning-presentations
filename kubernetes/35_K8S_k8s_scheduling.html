Scheduling Basics
What is Scheduling in Kubernetes?
⎈
The Scheduler decides where every Pod runs — when you create a Pod, it starts in "Pending" state. The scheduler finds the best node and assigns it.
🔍
Two-phase decision: Filter then Score — first eliminates unsuitable nodes (no resources, wrong labels), then scores remaining nodes and picks the best one.
📦
Based on resources, rules & constraints — checks CPU/memory availability, node labels, affinity rules, taints, and topology constraints before placing a Pod.
📈
Smart scheduling = better reliability — proper placement improves performance, prevents overloaded nodes, and enables high availability across zones.
Analogy: Think of the Scheduler as an airline seat allocator ✈️ — checks your requirements (window seat, extra legroom) against available seats and assigns the best match.

Why It Matters
Why Scheduling Rules Are Needed
⚡
GPU workloads need GPU nodes — an ML training job scheduled on a CPU-only node will either crash or run thousands of times slower. Labels + NodeSelector solve this.
🧠
Memory-intensive apps need high-RAM nodes — an in-memory database needs 256 GB RAM. Scheduling it on a 16 GB node causes OOM kills. Node Affinity targets the right class of node.
🔒
Secure/compliance nodes must be isolated — PCI or HIPAA workloads may only run on audited nodes. Taints prevent other Pods from sharing those nodes accidentally.
📍
HA requires spreading across zones — running all DB replicas on the same node means one node failure = full outage. Pod Anti-Affinity spreads them across availability zones.
Bottom line: Without scheduling control, Kubernetes places Pods randomly. You lose performance, security, and reliability. Scheduling rules = fine-grained workload control.

NodeSelector
What is NodeSelector?
🏷
Simplest scheduling method in Kubernetes — add a label to a Node, reference it in the Pod spec. Pod only lands on a node with that exact label.
📋
Works as a hard requirement (not a preference) — if no node has the matching label, the Pod stays in Pending state forever and never runs.
✅
Use case: basic workload separation — route GPU jobs to GPU nodes, route disk-heavy jobs to SSD nodes. Simple, effective, easy to understand.
⚠
Limitation: no flexibility — strict match only. No "prefer" option. Cannot use operators like >, <, or "In". For that, use Node Affinity.
YAML Snippet
spec:
  nodeSelector:
    gpu: "true"
  containers:
    - name: ml-trainer

Affinity
What is Affinity?
🧲
Advanced, flexible scheduling rules — goes beyond NodeSelector. Supports operators (In, NotIn, Exists, Gt), soft preferences, and Pod-to-Pod awareness.
📋
Required vs Preferred — requiredDuringScheduling is a hard rule (like NodeSelector). preferredDuringScheduling is a soft preference — Pod still schedules if no match.
🖥
Node Affinity — attract a Pod to specific nodes based on node labels. E.g. "must run in us-east-1, prefer SSD storage".
🤝
Pod Affinity / Anti-Affinity — place Pods near or far from other specific Pods. E.g. "co-locate web + cache" or "spread DB replicas across zones".
Key upgrade over NodeSelector: Affinity gives you "prefer" (best-effort) in addition to "require" (hard), making scheduling far more resilient and expressive.
Affinity Types
🖥 Node Affinity
Pod ↔ Node Labels
e.g. "run in us-east, on SSD nodes"
🤝 Pod Affinity
Pod ↔ Other Pod Labels
e.g. "place near cache Pod (same node)"
↔ Pod Anti-Affinity
Pod avoids other Pod Labels
e.g. "don't co-locate DB replicas"
All three can be combined in one Pod spec

Affinity Deep Dive
Node Affinity vs Pod Affinity vs Anti-Affinity
🖥
Node Affinity
Pod targets specific nodes
Attract a Pod to nodes with specific labels. Supports required (hard) and preferred (soft) rules. More powerful than NodeSelector.
Example: prefer SSD, must be us-east
💡 Use for: geo-routing, hardware classes, env tiers
🤝
Pod Affinity
Run near specific Pods
Schedule a Pod on the same node (or zone) as another Pod. Reduces network latency between co-dependent services. Example: Web app next to its Redis cache.

Taints & Tolerations
Taints & Tolerations — Restrict and Allow
🚫
Taints
Applied to Nodes — repel Pods
A taint marks a node as restricted. Regular Pods that don't have a matching toleration are blocked from scheduling on it. Protects special-purpose nodes.
Three taint effects:
NoSchedule — don't place new Pods here
PreferNoSchedule — avoid if possible
NoExecute — evict existing Pods too
kubectl taint nodes node1 gpu=true:NoSchedule
pair
🔑
Tolerations
Applied to Pods — bypass taints
A toleration gives a Pod the permission to run on a tainted node. It says: "I can handle this taint." The Pod is then scheduled normally on that node.
YAML Toleration:
tolerations:
- key: "gpu"
  operator: "Equal"
  value: "true"
  effect: "NoSchedule"
🔑 Without this toleration → Pod blocked. With it → Pod enters.

✅ Best Practices
◆Start with NodeSelector for simple cases
◆Upgrade to Affinity when you need "prefer"
◆Use Taints for dedicated / sensitive nodes
◆Anti-Affinity for HA across zones
◆Always label nodes clearly + consistently
⚠ Common Mistakes
✗Forget to label nodes before deploying
✗Tainting without matching tolerations
✗Too strict rules → Pods stuck Pending
