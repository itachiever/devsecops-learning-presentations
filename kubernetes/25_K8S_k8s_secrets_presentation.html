01 / 09
The Problem
Why Hardcoded Credentials are Dangerous
🔑
Apps need sensitive data to function — database passwords, API keys, tokens, and TLS certs are required at runtime
💀
Hardcoding is a security disaster — credentials in source code get committed to Git, shared with the team, and eventually leaked
📄
Plain text in ConfigMap is insecure — ConfigMap stores data unencrypted. Anyone with cluster read access can see credentials.
🐳
Secrets inside Docker images are exposed — anyone who pulls the image can extract credentials from the container layers
🔄
Rotating credentials is painful — changing a password means modifying code, rebuilding the image, and redeploying
The fix: Kubernetes Secrets store sensitive data encoded and separate from code — injected securely at runtime, never visible in source.
❌ Credentials Exposed — Dangerous
// app.js — DANGEROUS ❌
const DB_PASS = "mypassword123";
const API_KEY = "sk-abc123xyz";
🐙 Committed to GitHub = leaked
🔍 Visible in image layers
💀 Anyone who pulls image can read it
✅ With Kubernetes Secrets
🔐 Secret: encoded + access-controlled
↓ inject at runtime
📦 Pod gets credentials securely
✅ Source code has zero credentials

Definition
What is a Kubernetes Secret?
🔐
Kubernetes object for storing sensitive data — passwords, tokens, keys, and certificates stored separately from application code
🛡
Data is Base64 encoded — values are encoded so they can be safely transported. Not plain text like ConfigMap.
📦
Used securely by Pods — injected as environment variables or mounted as files. Container receives decoded value at runtime.
🧬
Similar to ConfigMap but for sensitive data — same injection mechanisms, but with RBAC access controls and encoding by default
🔑
Access-controlled via RBAC — only authorized Pods and service accounts can read a Secret. Unauthorized access is blocked.
Important: Base64 is encoding, not encryption. Enable etcd encryption-at-rest or use HashiCorp Vault for full protection.
Secret YAML
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  DB_PASS: dGhpcyBpcyBlbmNvZGVk
  API_KEY: a2V5LXZhbHVlLWhlcmU=
base64 encoded
→
decoded in Pod
✅ Credentials never in source code

Secure Injection Flow
How Secrets Work — Step by Step
1
Admin creates Secret with encoded values
Values are base64-encoded. Applied via kubectl apply or kubectl create secret. Never committed to Git.
🔐
↓
2
Secret stored securely in cluster etcd
Stored in etcd. With encryption-at-rest enabled, it is encrypted on disk. RBAC controls who can read it.
🗄
↓
3
Pod spec references Secret by name
Pod YAML uses envFrom secretRef or volume with secret source. Kubernetes authorizes via RBAC and fetches it.
📦
↓
4
Secret decoded and injected into container
Kubernetes decodes base64. Container receives plain-text credentials as an env var or file at a mount path.
💉
↓
5
App reads decoded credentials at runtime
App reads process.env.DB_PASS — plain text available only at runtime. Never hardcoded anywhere. ✅
✅

Injection Methods
3 Ways to Use Secrets in a Pod
🌿
Method 1 — Env Variables
All secret keys become env vars
⭐ Popular
# Pod YAML
envFrom:
  - secretRef:
    name: db-secret
# Result inside container:
$DB_PASS = decoded_password
All keys → decoded env vars ✅
🎯
Method 2 — Specific Keys
Selective secret injection
# Pick one specific key
env:
  - name: DATABASE_PASS
    valueFrom:
      secretKeyRef:
        name: db-secret
        key: DB_PASS
Inject only needed keys ✅
📁
Method 3 — Mounted Files
TLS certs & SSH keys as files
# Mounted at /etc/secrets/
volumes:
  - name: tls-certs
    secret:
      secretName: tls-secret
# /etc/secrets/tls.crt ✅
TLS certs & SSH keys ✅

Real-World Example
App Connecting to DB Using a Secret
🗄
Scenario: A Node.js API needs to connect to a PostgreSQL database in production
🔐
Admin creates Secret: DB password base64-encoded in db-secret.yaml. Applied to cluster. Never committed to Git.
📦
Pod references Secret: Pod YAML uses secretRef: db-secret. Kubernetes authorizes via RBAC and injects it.
📱
App reads decoded password: process.env.DB_PASS — clean decoded value at runtime. Zero hardcoding.
🔄
Password rotation is easy: Update the Secret, restart the Pod, new credentials loaded — zero code changes, zero rebuilds.
Security win: DB password exists in one place — the Kubernetes Secret. Not in code, not in images, not in logs.
