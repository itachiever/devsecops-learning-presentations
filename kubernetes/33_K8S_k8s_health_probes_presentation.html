The Problem
Why Health Checks Are Critical
🧟
Container running ≠ App working — a container can appear healthy while the app inside is frozen, deadlocked, or returning errors to every request
😴
Apps can freeze without crashing — a Java app with a thread deadlock, a Node app stuck in an infinite loop — the process runs but serves nothing
🚀
Apps take time to start up — Spring Boot, databases, and heavy services may need 30-60 seconds before they're ready. Sending traffic too early = errors.
👥
Users hit broken endpoints — without health checks, a Service may route user traffic to a Pod that's completely broken. Bad user experience.
🤖
Kubernetes needs to know app state — probes give K8s the intelligence to restart broken apps and withhold traffic from unready ones automatically
Without probes: K8s sends traffic to broken Pods and never restarts frozen apps. Users see errors forever. With probes: automatic detection + recovery.

Definition
What are Probes?
🔬
Probes are periodic health checks — Kubernetes runs them on a schedule (every N seconds) to continuously verify the state of each container
🤖
Kubelet on each node runs the probes — not an external service. The agent on the node directly checks the container and reports the result
📊
Three probe results: Success (container is healthy), Failure (container is unhealthy), Unknown (check couldn't complete)
⏱
Configurable timing parameters — initialDelaySeconds, periodSeconds, failureThreshold, successThreshold
🛡
Failure threshold prevents flapping — probe must fail N times in a row (default 3) before action is taken. Single glitch won't trigger restart.
Analogy: Probes = doctor checking patient vitals every 10 seconds 🏥. Not just once — continuously. If pulse fails 3 checks in a row → take action.
Probe Parameters

  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3
  successThreshold: 1
⏱ initialDelay → wait before first check
🔄 period → how often to check
❌ failureThreshold → fails before action

Liveness Probe
Is the App Alive?
💚
Detects frozen or dead applications — checks if the app is still functioning. A deadlocked app, infinite loop, or hung process = liveness failure.
🔄
Failure → Container is restarted — Kubernetes kills the container and starts a fresh one. Like rebooting a frozen computer automatically, no human needed.
⏱
Runs continuously throughout Pod lifetime — not just at startup. Checks every N seconds forever. A healthy app can become unhealthy over time.
⚠
Use initialDelaySeconds carefully — if set too low, K8s may restart the app before it even finishes starting up. Match your app's startup time.
🎯
Checks a simple endpoint — /healthz — the endpoint should return 200 if the app is functional. Not a deep health check — just "am I running OK?"
Analogy: Liveness = manager checking if employee is awake and conscious 👀. Not working yet — just alive. If they won't respond → call ambulance (restart).

Self-healing — no human intervention ✅

Readiness Probe
Is the App Ready for Traffic?
🔵
Checks if the app can handle requests — even a live app may not be ready yet. DB connection not established, cache warming up, startup initialization in progress.
🚦
Failure → Pod removed from Service endpoints — the Pod stays running but the Service stops routing traffic to it. Users no longer hit this Pod.
🤝
Pod is NOT restarted on readiness failure — unlike liveness, a readiness failure is not a restart signal. The container continues but just receives no traffic.
🔄
Re-added to endpoints when ready — once readiness passes again (recovery, cache loaded, DB reconnected), traffic automatically flows back to this Pod.
🚀
Critical for rolling updates — during deployment, new Pods only receive traffic once they pass readiness. Prevents users from hitting uninitialized instances.
Analogy: Readiness = manager checking if employee is ready to serve customers 🛎. Awake but still having coffee? Don't open the counter yet. Wait till ready.
Readiness Flow
👥 User requests
↓ via Service
🔵 GET /ready → 200?
✅ Ready
In endpoints
Gets traffic
⚠ Not Ready
Out of endpoints
No traffic
Container stays alive — just no traffic!

Probe Methods
3 Ways to Check — Probe Types
🌐
HTTP GET
Most common · web apps
⭐ Popular
K8s sends an HTTP GET to a specified path and port. If it returns 2xx or 3xx → Success. 4xx/5xx or timeout → Failure.
httpGet:
  path: /healthz
  port: 8080
  scheme: HTTP
✅ Use for REST APIs & web apps
🔌
TCP Socket
Non-HTTP services
K8s tries to open a TCP connection to the specified port. If connection succeeds → Success. Connection refused → Failure. No HTTP needed.
tcpSocket:
  port: 5432

# Great for databases
✅ Use for databases, gRPC, Redis
⌨️
Exec Command
Custom scripts
K8s runs a command inside the container. Exit code 0 → Success. Any non-zero exit → Failure. Most flexible, can run any logic.
exec:
  command:
  - cat
  - /tmp/healthy
✅ Use for custom health logic
Which to Use?
🌐 HTTP GET
REST APIs, web servers
🔌 TCP Socket
MySQL, Redis, Kafka
⌨ Exec
Custom scripts, file checks
All work with both Liveness & Readiness

Best Practices
Health Probe Best Practices
⏱
Set initialDelay Correctly
Match your app's startup time. If Spring Boot takes 20s, set initialDelaySeconds: 30. Too short = false restarts.
🪶
Keep Probe Endpoints Lightweight
/healthz should just return 200 instantly. Never run DB queries in a liveness probe — that creates false failures under load.
🔵
Readiness Can Be Deep
Readiness can check DB connectivity, cache status, downstream services. If dependencies are down, go not-ready to avoid serving bad responses.
🔢
Set Failure Threshold > 1
Default is 3 consecutive failures. Never set it to 1 — a single network hiccup or slow response will trigger false restarts on healthy apps.
🚫
Don't Liveness-Probe Dependencies
Never check external services in liveness probes. If the DB is down, that's not the app's fault — a liveness failure would restart your healthy app unnecessarily.
🚀
Use Startup Probe for Slow Apps
startupProbe — a third probe type. Disables liveness until startup succeeds. Perfect for legacy apps with long boot times.
