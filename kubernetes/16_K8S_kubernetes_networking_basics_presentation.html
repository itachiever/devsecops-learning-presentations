The Foundation
Why Networking Matters in K8s?
📦
Your app is split across many Pods — frontend, backend, database all in separate Pods that need to talk
🗣
Pods must communicate reliably — the web Pod needs to call the API Pod, which calls the database Pod
👤
Users need to reach your app — external traffic from browsers and mobile apps must get in to your Pods
🔗
Networking connects everything — without it, your microservices are islands that can't work together
Think of it: Networking = the roads in a city. Without roads (network), nothing reaches anyone.

Key Concept
Pod-to-Pod Communication
🪪
Every Pod gets a unique IP address — automatically assigned when the Pod starts. No manual config.
🤝
Pods communicate directly — any Pod can send a request to any other Pod using its IP address
🚫
No NAT required — NAT = Network Address Translation. K8s doesn't hide Pod IPs. Direct routing.
🌐
Works across nodes — Pod on Node 1 can directly reach Pod on Node 2 as if they're on the same machine
⚠ But: Pod IPs change when Pods restart. That's why we need Services for stable access.

Networking Model
The Flat Network Model
📐
All Pods live in one flat network — regardless of which node they're on, they share the same virtual network space
📡
Any Pod can reach any other Pod — using its IP directly, no firewalls, no NAT between them by default
⚙️
No manual routing config needed — the CNI plugin (e.g., Calico, Flannel) handles all of this automatically
🏠
Like one giant LAN — Local Area Network. All Pods behave as if they're plugged into the same switch
K8s networking rule: Every Pod can communicate with every other Pod without NAT — this is guaranteed by the Kubernetes networking model

Abstraction Layer
Role of Services in Networking
⏳
Pods are temporary — IPs change — every time a Pod restarts it gets a new IP. You can't hardcode Pod IPs.
🏷
Services provide a stable address — Services get a fixed IP (ClusterIP) and DNS name that never changes
🎯
Fixed entry point to your Pods — other services always call the Service, never the Pod IP directly
⚖️
Load balances traffic — if there are 3 Pods behind a Service, traffic is spread evenly across all 3
Analogy: Service = a reception desk 🏢. You call reception (stable), they connect you to the right employee (Pod).

Name Resolution
DNS in Kubernetes
🌐
Kubernetes has built-in DNS — automatically runs CoreDNS inside the cluster so Pods can use names
📛
Pods can use Service names — instead of 10.96.0.10, just write api-svc
🧠
No need to remember IPs — DNS resolves names to IPs automatically. Apps use names, DNS does the work.
📍
Full DNS format — service-name.namespace.svc.cluster.local — but short names also work
Analogy: K8s DNS = phonebook 📖. You look up "api-svc" and get back 10.96.0.10. You use the name, not the number.

Under the Hood
Networking Components (High-Level)
🔀
kube-proxy — runs on every node, maintains iptables/IPVS rules so traffic is routed to the correct Pod via Service
🔌
CNI (Container Network Interface) — plugin that actually creates the network for Pods (e.g., Calico, Flannel, Cilium)
🌐
CoreDNS — the internal DNS server in K8s. Watches Services and keeps DNS records up to date
⚙️
Services — virtual IP abstraction. Not a real process, just an IP/DNS entry + routing rules
You don't configure these manually — they're set up when you create a Kubernetes cluster. As a user, you interact with Services and DNS names.
