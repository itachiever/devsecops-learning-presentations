Observability
Why Logging is Important
📋
Applications generate logs constantly — every request, error, startup event, and warning is recorded. Logs are your app's real-time diary.
🔥
Logs reveal failures and root causes — when something breaks at 2 AM, logs tell you exactly what crashed, when, and why. Without logs, debugging is guesswork.
🔍
Essential for troubleshooting — trace an error back through hundreds of microservices. Logs connect the dots between what the user saw and what the system did.
📈
Understand application behavior — monitor response times, traffic patterns, and usage trends. Logs help you optimize before problems occur.
🏭
Mandatory in production environments — no serious production system runs without logging. Compliance, auditing, and SLA monitoring all depend on it.
Analogy: Logs are like a flight black box ✈️ — when something goes wrong, investigators replay the recorded events to find exactly what happened and why.

Definition
What Are Logs?
📄
Logs are timestamped records of events — every action your application takes is written as a log entry: timestamp, severity level, and a message describing what happened.
🔴
ERROR logs — something failed. Database connection refused, null pointer exception, HTTP 500. These need immediate attention.
🟡
WARN logs — potential problems. High memory usage, deprecated API call, slow query. Not broken yet, but watch closely.
🟢
INFO logs — normal events. "Server started on port 8080", "User logged in", "Order processed". The app's narrative of its own behavior.
🔵
DEBUG logs — detailed technical traces for developers. Variable values, function call sequences. Usually disabled in production.
Sample Log Stream
INFO 2024-01-15 09:00:01 Server started :8080
INFO 09:00:14 GET /api/users 200 12ms
WARN 09:01:32 DB pool at 85% capacity
WARN 09:01:45 Slow query: 1.2s on users
ERROR 09:02:01 DB connection refused
ERROR 09:02:01 NullPointerException line 47
DEBUG 09:02:02 retry attempt 1/3
🟢 INFO — normal ops
🟡 WARN — watch it
🔴 ERROR — act now
🔵 DEBUG — dev only

How It Works
Logging in Kubernetes
📦
Containers write to stdout and stderr — this is the Kubernetes logging convention. Your app prints logs normally; Kubernetes intercepts and saves them.
🖥
Kubelet saves logs to the node filesystem — each node stores its container logs at /var/log/containers/ as rotated log files.
🔍
View logs instantly with kubectl — kubectl logs <pod-name> fetches live logs from any Pod. Use -f to stream continuously.
📋
Multi-container Pods — use -c <container-name> to specify which container's logs to fetch when a Pod has multiple containers.
⚠
Logs are lost when a Pod is deleted — kubectl logs only works while the Pod exists. Restarted or deleted Pods lose their logs. This is why centralized logging matters.

The Problem
Challenges in Kubernetes Logging
⏱
Pods are ephemeral — they come and go — a crashed Pod is deleted and replaced. Its logs vanish with it. kubectl logs only works on running Pods.
🔄
Containers restart constantly — CrashLoopBackOff, OOMKills, liveness probe failures. Each restart wipes the previous log. Use --previous flag, but it's limited.
🌐
Distributed logs across hundreds of Pods — one request might touch 10 microservices across 10 Pods on 5 nodes. Finding the bug means manually querying each one.
💾
Node disk is limited — logs are rotated and truncated on the node. High-volume services fill disk fast. Old logs get deleted automatically.
Bottom line: kubectl logs alone is not enough for production. You need a system that collects logs before Pods disappear and stores them centrally where you can search them.

The Solution
Centralized Logging
🔄
A log agent runs on every node — deployed as a DaemonSet (one Pod per node), agents like Fluentd or Fluent Bit continuously ship logs before Pods disappear.
🗄
All logs stored in a central system — Elasticsearch or Loki receives logs from all nodes. One place to search across all Pods, all nodes, all time.
🔍
Powerful search and filtering — query by Pod name, namespace, log level, timestamp, or any keyword. Find that one error across a million log lines in milliseconds.
📊
Visualize with dashboards — Kibana or Grafana turns raw logs into charts, alerts, and dashboards. Set up alerts for ERROR spikes or anomaly detection.
Key benefit: Even if a Pod crashes and disappears, its logs are already safe in the central store. Search history, correlate events, and debug without touching Pods at all.

Best Practices
Logging Best Practices
🏗
Always use centralized logging — never rely solely on kubectl logs in production. Deploy a log agent (Fluent Bit DaemonSet) from day one, even in dev clusters.
📐
Use structured logs (JSON format) — instead of plain text like "User logged in", output {"level":"info","user":"u123","action":"login"}. JSON is machine-readable and easily filtered.
🏷
Include context in every log line — log the request ID, user ID, Pod name, and operation. This enables end-to-end tracing across multiple microservices.
🔔
Set up log-based alerts — alert on ERROR rate spikes, specific exception patterns, or security events. Don't wait for users to report problems.
🔒
Redact sensitive data — never log passwords, credit card numbers, or PII. Use log scrubbing in Fluentd/Fluent Bit before shipping. Compliance (GDPR, PCI) requires this.
Golden rule: Logs should tell you what happened, when it happened, where it happened, and enough context to know why.
✅ Good Log (Structured)
{
  "level": "error",
  "msg": "DB timeout",
  "pod": "api-7f9b",
  "requestId": "abc-123",
  "duration": 3021
}
❌ Bad Log (Unstructured)
Something went wrong
user=john@email.com
password=abc123 ← ⚠️ EXPOSED!
No context · exposes PII · unsearchable
