The Need
Why Applications Need Identity
🤖
Applications often need to talk to Kubernetes — CI/CD pipelines deploy apps, monitoring tools read metrics, operators manage resources — all from inside the cluster
📖
Pods may need to read cluster data — a monitoring app needs to list Pods, a logging agent needs to read ConfigMaps, an autoscaler checks resource usage
🔐
Kubernetes API requires authentication — any request to the K8s API must include a valid identity. Without one, the request is rejected with 401 Unauthorized.
❌
Human user credentials are wrong for apps — using Alice's kubeconfig inside a Pod is dangerous, hard to rotate, and breaks when Alice leaves
✅
Service Accounts = machine identity — dedicated, automatically managed credentials for applications. No human involved. Scoped and revokable.
Rule: Never put human credentials inside a Pod. Always use a dedicated ServiceAccount with minimum required RBAC permissions.
❌ Using Human Credentials in Pod
🔑 alice's kubeconfig in Pod env
💀 Alice leaves → app breaks
🔓 Admin access in production Pod
🔄 Hard to rotate credentials
✅ Using Service Account
🤖 Dedicated SA per application
🔄 Token auto-rotated by K8s
🎯 Least-privilege RBAC roles
🛡 Independent of humans
02 / 09
Definition
What is a Service Account?
🤖
A Kubernetes identity for workloads — a ServiceAccount is a namespaced K8s object that represents the identity of a Pod or process running inside the cluster
🎫
K8s automatically provides a JWT token — when a Pod is assigned a ServiceAccount, K8s mounts a signed token into the Pod at a standard path
👤
Different from human user accounts — ServiceAccounts are for machines/apps, managed by Kubernetes itself, not external identity providers
📦
Scoped to a namespace — a ServiceAccount in "monitoring" namespace is separate from one in "prod". Isolation by design.
🔗
Works with RBAC — assign Roles to ServiceAccounts using RoleBindings, just like you would for a human user. Fine-grained access control.
Analogy: ServiceAccount = employee badge for an automated system 🤖. It identifies the robot, not a human. And it only opens the doors it's permitted to.
ServiceAccount YAML
apiVersion: v1
kind: ServiceAccount
metadata:
  name: monitor-sa
  namespace: monitoring
↓ assigned in Pod spec
spec:
  serviceAccountName: monitor-sa
K8s auto-mounts token into Pod ✅

Default Behaviour
The Default Service Account
🏠
Every namespace has a "default" ServiceAccount — Kubernetes creates it automatically when a namespace is created. It always exists.
📦
Pods use it automatically if none specified — if you don't set serviceAccountName, K8s assigns the "default" SA automatically
⚠
Default SA has very limited permissions — by default it can only do basic operations. You need to bind Roles to it explicitly for more access.
❌
Don't rely on default SA in production — if you add permissions to the default SA, ALL Pods in that namespace inherit them. Security risk.
✅
Create dedicated SAs per application — one SA per workload with only the permissions that specific workload needs. Separation of concerns.
Best practice: Never add permissions to the "default" ServiceAccount. Create a named SA per app with exactly the permissions it needs.

06 / 09
Access Control
RBAC with Service Accounts
🤖
Service Accounts are RBAC subjects — just like human users, SAs can be bound to Roles and ClusterRoles using RoleBindings
📋
Create a Role with minimum permissions — define exactly what verbs and resources the application needs. Nothing more, nothing less.
🔗
Bind the Role to the ServiceAccount — RoleBinding with subject kind: ServiceAccount links them. K8s enforces on every API call.
🎯
Fine-grained per-application access — monitoring app reads pods only, CI/CD creates deployments only, backup app reads volumes only
🛡
Compromised Pod = limited blast radius — if a Pod is breached, attacker gets only what that SA is allowed. Not the whole cluster.
Pattern: 1 App = 1 ServiceAccount = 1 Role = specific permissions. Never share SAs between different applications.

Real-World Example
Monitoring App Securely Reading Cluster Data
📊
Scenario: Prometheus monitoring needs to discover and scrape metrics from all Pods in the cluster
🤖
Step 1 — Create SA: prometheus-sa in the "monitoring" namespace
📋
Step 2 — Create ClusterRole: "prometheus-reader" allows get/list/watch on pods, services, endpoints cluster-wide
🔗
Step 3 — ClusterRoleBinding: links prometheus-sa to prometheus-reader. Prometheus can now discover all Pods.
📦
Step 4 — Pod uses SA: Prometheus Pod spec sets serviceAccountName: prometheus-sa. Token auto-mounted. Scraping works.
🛡
Result: If Prometheus is compromised, attacker can only read Pod/service data — cannot delete, create, or access Secrets. Contained.
Prometheus RBAC Flow
📦 Prometheus Pod
serviceAccountName: prometheus-sa
↕ identity via
🤖 prometheus-sa
namespace: monitoring
↕ bound to
📋 ClusterRole: prometheus-reader
get/list/watch pods, services, endpoints
✅ GET /api/v1/pods → 200 OK
✅ GET /api/v1/services → 200 OK
❌ DELETE /api/v1/pods → 403
❌ GET /api/v1/secrets → 403
