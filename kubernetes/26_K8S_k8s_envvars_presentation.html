The Problem
Why Hardcoded Values Don't Work
🔒
Hardcoding config makes apps inflexible — database URLs, API endpoints, ports baked into code. Changing one value means changing the code.
🌍
Different environments need different settings — dev uses localhost, staging uses test DB, prod uses cloud DB. Same code cannot serve all three.
🔄
Every config change = rebuild + redeploy — update a single URL, rebuild the Docker image, re-push to registry, redeploy. Slow and error-prone.
🐛
Wrong values go to wrong environments — a dev accidentally points to the production database. Config buried in code is easy to get wrong.
🔐
Sensitive values exposed in code — passwords and tokens hardcoded in source files get committed to Git and shared with the whole team
Solution: Environment Variables inject configuration at runtime. Change settings without touching the code or rebuilding the image.
❌ Hardcoded — Inflexible
// app.js — HARDCODED ❌
const DB = "localhost:5432";
const PORT = 8080;
Change env? Rebuild image 🔄
✅ Env Vars — Flexible
// reads from environment
const DB = process.env.DB_URL;
const PORT = process.env.PORT;
Change env? Update YAML only ✅
02 / 09
Definition
What are Environment Variables?
⚡
Key-value pairs passed to applications — each variable has a NAME and a VALUE. e.g. DB_HOST=postgres.svc
💉
Configuration injected dynamically at runtime — your container starts and receives its settings through env vars. No rebuild needed.
📖
Application reads values during execution — via process.env (Node), os.getenv() (Python), System.getenv() (Java)
🌍
Universal pattern across all languages — every programming language supports reading environment variables. Industry-standard approach.
🔗
Can come from multiple sources — defined directly in YAML, loaded from ConfigMaps, or injected securely from Kubernetes Secrets
Analogy: Env vars = a briefing given to an employee before they start work 📋. Same employee, different briefings for different jobs.
KEY = VALUE Format
DB_HOST=postgres.prod.svc
DB_PORT=5432
LOG_LEVEL=info
APP_ENV=production
MAX_CONN=100
↓ read by app at startup
📱 App reads:
process.env.DB_HOST
os.getenv("LOG_LEVEL")
✅ No hardcoding anywhere

Kubernetes Injection Flow
How Environment Variables Work in Kubernetes
1
Env vars defined in Pod YAML spec
Under the env: or envFrom: field in the container spec. Written once, applied every time Pod starts.
📄
↓
2
Kubernetes schedules the Pod
K8s scheduler selects a node. Kubelet on that node receives the Pod spec including all env var definitions.
☸
↓
3
K8s resolves ConfigMap and Secret sources
If env vars come from ConfigMap or Secret references, Kubernetes fetches and decodes those values automatically.
🔗
↓
4
Container starts with env vars injected
Container process starts with all env vars set. App reads them instantly from the process environment. No extra setup.
📦
↓
5
Application uses config dynamically
App connects to the correct DB, uses the right API endpoint, runs at the correct log level — all from env vars. ✅
✅
Pod YAML Example
spec:
  containers:
  - name: my-app
    env:
    - name: DB_HOST
      value: "db.svc"
    - name: LOG
      valueFrom:
        configMapKeyRef:
          name: app-cfg
          key: LOG_LEVEL
Container gets all vars at start ✅

Configuration Sources
Where Do Environment Variables Come From?
📄
Source 1 — Direct in YAML
Simplest
Hard-coded in the Pod spec. Good for static values that don't change between envs.
name: PORT   value: "8080"
⚙
Source 2 — ConfigMap
⭐ Popular
Non-sensitive settings stored externally. Reusable across multiple Pods and environments.
configMapKeyRef: app-config
🔐
Source 3 — Secret
Secure
Sensitive credentials injected securely. Decoded automatically. Access controlled via RBAC.
secretKeyRef: db-secret
☸
Source 4 — K8s Auto-Generated
Kubernetes automatically injects cluster info — Service IPs, ports, namespace, node name — into every Pod.
MY_SVC_SERVICE_HOST=10.0.0.1

Integration
ConfigMap + Secret → Env Variables
⚙
ConfigMap provides non-sensitive settings — database host, port, log level, API URLs, feature flags — all safe to expose
🔐
Secret provides sensitive credentials — database password, API tokens, TLS keys — encoded and RBAC-controlled
💉
Both injected as environment variables — application sees a unified set of env vars. It doesn't know or care where they came from.
🎯
Centralized configuration management — change one ConfigMap or Secret → all Pods using it get the update. One source of truth.
🔄
Works across all environments — dev uses dev-configmap + dev-secret, prod uses prod versions. Same app, same YAML structure.
Best practice: Always use ConfigMap for non-sensitive and Secret for sensitive data. Never mix credentials into ConfigMap.

Benefits
Why Use Environment Variables?
🔓
Flexible configuration without code changes — change DB host, port, log level — all without modifying a single line of application code
🌍
Easy multi-environment management — same Docker image, different env vars per environment. Dev → Test → Prod without rebuilding.
🚀
No image rebuild required — update a ConfigMap or env var YAML, restart the Pod — new config is live in seconds. No CI/CD pipeline run needed.
🚢
Better application portability — bring your container to any cluster, any cloud provider — just set the right env vars. Works everywhere.
🎯
Simplifies deployments and rollbacks — configuration lives in version-controlled YAML files. Roll back config = roll back YAML. Simple and auditable.
12-Factor App: Storing config in environment variables is factor III of the 12-Factor App methodology — the industry standard for modern cloud-native applications.

Real-World Example
Full Config Flow — App Connecting to Database
⚙
ConfigMap stores non-sensitive DB settings: DB_HOST=postgres.prod.svc, DB_PORT=5432, DB_NAME=orders
🔐
Secret stores credentials: DB_USER=admin and DB_PASS=••• — base64 encoded, RBAC protected
💉
Pod YAML references both: envFrom: configMapRef + secretRef — K8s merges all vars into container
📱
App builds DB connection string at runtime: reads all five vars, constructs postgres://admin:pass@host:5432/orders
✅
Switch to production: swap to prod-configmap + prod-secret → new DB, same app, zero code changes, zero rebuilds
