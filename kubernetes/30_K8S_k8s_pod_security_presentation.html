The Problem
Why Pod Security Matters
📦
Containers are powerful but dangerous by default — without restrictions, a container can access the host filesystem, run as root, and escape to the underlying node
💀
Misconfigured containers become attack vectors — a vulnerable app running as root inside a container gives attackers host-level access if exploited
🔓
Excessive permissions = large attack surface — container with NET_ADMIN capability can reconfigure host networking. That's way more than a web app needs.
🚪
Container escape is a real threat — a privileged container can break out of its isolation and compromise the entire node and cluster
🛡
Security Context solves this — define exactly what the container is allowed to do. Block everything else. Defense by configuration.
Principle: Give containers the minimum permissions they need. Deny everything else. A web server does NOT need to modify system files or access the host kernel.
❌ Insecure Container Default
🔓 Runs as root (UID 0)
💣 Can write anywhere on filesystem
⚡ All Linux capabilities inherited
🚪 Can escalate privileges
✅ Hardened Container
👤 Runs as UID 1000 (non-root)
🔒 Filesystem is read-only
⚡ capabilities: drop ALL
🛡 allowPrivilegeEscalation: false

Definition
What is a Security Context?
⚙
Security settings for Pods and containers — a field in the Pod/container spec that controls OS-level security behavior: user, capabilities, filesystem, privileges
📦
Applied at two levels — spec.securityContext affects the whole Pod; containers[].securityContext per container
🎯
Controls permissions and access levels — WHO the process runs as, WHAT files it can write to, WHAT kernel capabilities it has
🔒
Enforced by the Linux kernel — not just configuration labels. The OS actually blocks operations the Security Context forbids.
🌍
Universal — works across all container runtimes — Docker, containerd, CRI-O all honor Security Context settings from Kubernetes
Analogy: Security Context = office access rules 🏢. Employee can enter their floor, use their computer, but cannot access the server room or finance files.
Security Context Scope
📦 Pod-level securityContext
securityContext:
  runAsUser: 1000
  runAsGroup: 3000
  fsGroup: 2000
Applies to ALL containers in Pod
📱 Container-level securityContext
securityContext:
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop: ["ALL"]
Overrides Pod-level for this container
03 / 10
User Identity
Running as Non-Root
💀
Root = UID 0 = full OS access — a process running as root inside a container has the same power as root on the host (in misconfigured setups)
👤
runAsUser: 1000 — specify a non-privileged UID. Container runs as a regular user, not root. Dramatically reduces breach impact.
🚫
runAsNonRoot: true — Kubernetes will refuse to start the container if the image tries to run as root. Hard enforcement, not just a suggestion.
👥
runAsGroup: 3000 — set the group ID for the container process. Controls which files and resources the process can access by group ownership.
🗂
fsGroup: 2000 — all files created in volumes owned by this GID. Essential when containers need to read/write shared volumes.
Analogy: Root in a container = master key to the building 🗝. Non-root = regular employee key that opens only their office. Much safer.
❌ Running as Root
securityContext:
  # nothing specified
  # defaults to root (UID 0)
🔓 Can read /etc/passwd
💀 Can install backdoors
🚪 Can escape container
✅ Running as Non-Root
securityContext:
  runAsUser: 1000
  runAsNonRoot: true
  runAsGroup: 3000
✅ Limited OS access
✅ Cannot install system packages
✅ Contained breach radius

Privilege Levels
Privileged vs Restricted Containers
💣
Privileged Container
privileged: true
⚠ Avoid
Gets almost root access to the host. Can see all host processes, mount host filesystems, bypass cgroups. Essentially no isolation.
securityContext:
  privileged: true ❌
❌ Can mount host filesystems
❌ Can access host network devices
❌ Container escape trivial
vs
🛡
Non-Privileged Container
privileged: false (default)
⭐ Recommended
Isolated from host. Cannot access host resources unless explicitly permitted. Default for almost all workloads. Use privileged only for node-level tools.
securityContext:
  privileged: false ✅
  allowPrivilegeEscalation: false ✅
✅ Isolated from host filesystem
✅ Cannot escalate permissions
✅ Breach stays inside container
⚠ When is Privileged OK?
🔧 Node-level DaemonSets
📊 System monitoring agents
🌐 Network drivers (CNI)
💾 Storage drivers (CSI)
These are exceptions, NOT the rule
05 / 10
Linux Capabilities
Drop Unnecessary Capabilities
⚡
Linux capabilities are granular root powers — instead of "all or nothing" root, Linux splits root into ~40 specific capabilities (NET_ADMIN, SYS_PTRACE, etc.)
⚠
Containers inherit capabilities by default — a default container gets a large set of capabilities inherited from the container runtime. Too many for most apps.
🗑
Drop ALL, then add only what's needed — capabilities.drop: ["ALL"] removes everything, then add: ["NET_BIND_SERVICE"] only what you need
🌐
Most apps need zero extra capabilities — a typical web server, API, or worker only needs to run on a port. No special capabilities required at all.
🔧
Common legitimate capabilities — NET_BIND_SERVICE (ports <1024), SYS_TIME (set clock), rarely needed in practice
Rule: Start by dropping ALL capabilities. Add specific ones only if your app truly needs them. Most web apps need zero.
Capabilities YAML
securityContext:
  capabilities:
    drop:
      - ALL # remove everything
    add:
      - NET_BIND_SERVICE # only this one
❌ SYS_ADMIN — dropped
❌ NET_RAW — dropped
❌ SYS_PTRACE — dropped
✅ NET_BIND_SERVICE — kept
06 / 10
Filesystem Protection
Read-Only Filesystem & Resource Protection
🔒
readOnlyRootFilesystem: true — makes the container's root filesystem immutable. App can read files but cannot write or modify them during runtime.
🛡
Prevents runtime tampering — attackers who compromise the app cannot drop malicious files, modify binaries, or install backdoors on disk
📁
Use emptyDir for writable temp space — if your app needs to write (logs, temp files), mount a writable emptyDir volume at that specific path only
🚫
allowPrivilegeEscalation: false — prevents the process from gaining more privileges than it started with. Blocks setuid binaries and sudo inside the container.
🏠
seccompProfile: RuntimeDefault — applies a default set of syscall restrictions. Limits which kernel calls the container can make. Reduces kernel attack surface.
Security gain: A read-only FS + no privilege escalation means even if an attacker gets code execution inside the container, they cannot persist or escalate.
❌ Writable FS — Dangerous
💀 Attacker writes /tmp/backdoor.sh
💀 Modifies /etc/cron to persist
💀 Replaces app binary with malware
✅ Read-Only FS — Safe
securityContext:
  readOnlyRootFilesystem: true
volumeMounts:
- mountPath: /tmp
  name: tmp-dir # writable
✅ Cannot write to / — only /tmp
🏆 Gold standard
readOnly FS + non-root + drop ALL caps

Kubernetes Built-in Policy
Pod Security Standards (PSS) — 3 Tiers
💣
Privileged
No restrictions
⚠ Avoid
Completely unrestricted. Allows privileged containers, any user, all capabilities. Used only for system-level workloads like CNI plugins.
❌ Privileged containers allowed
❌ Root user allowed
❌ All capabilities allowed
Use for: kube-system, CNI, CSI
⚖
Baseline
Minimal restrictions
🏭 Common
Prevents known privilege escalations. Good starting point for most workloads. Blocks the most dangerous options while remaining practical.
❌ No privileged containers
⚠ Root allowed (discouraged)
✅ Dangerous caps blocked
Use for: most app namespaces
🛡
Restricted
Heavily hardened
⭐ Best
Follows security best practices strictly. Maximum isolation. Requires non-root, read-only FS, dropped caps. Some apps may need adjustments.
✅ Non-root enforced
✅ No privilege escalation
✅ Seccomp required
Use for: prod, sensitive workloads
How to Apply PSS
# Namespace label:
labels:
 pod-security.k8s.io/
 enforce: restricted
Applied via namespace labels
warn — show warnings
audit — log violations
enforce — block violations

Real-World Example
Production-Hardened API Container
🌐
Scenario: A Node.js REST API serving production traffic needs to be hardened for security
👤
runAsUser: 1001 — app runs as non-root user. Cannot access system files. Breach radius is contained.
🔒
readOnlyRootFilesystem: true — attacker cannot write malware to disk even if they gain code execution. Only /tmp is writable via emptyDir.
⚡
capabilities drop ALL, add NET_BIND_SERVICE — app binds to port 80 (needs NET_BIND_SERVICE). Every other capability dropped.
🛡
allowPrivilegeEscalation: false — even if a bug triggers sudo or setuid inside the container, it cannot gain more privileges. Hard blocked.
Net result: A full RCE exploit in this app cannot install persistence, escalate, or touch other Pods. Blast radius = only the app process. This is the goal.
Hardened YAML — Full
spec:
 securityContext:
  runAsUser: 1001
  runAsNonRoot: true
  fsGroup: 2000
 containers:
 - securityContext:
   allowPrivilegeEscalation: false
   readOnlyRootFilesystem: true
   capabilities:
    drop: ["ALL"]
    add: ["NET_BIND_SERVICE"]
   seccompProfile:
    type: RuntimeDefault
✅ PSS Restricted — compliant
