The Problem
Why One LoadBalancer Each Doesn't Scale
💰
One LoadBalancer per Service is expensive — cloud providers charge per LB. 10 services = 10 bills every month
🌐
Multiple external IPs are confusing — users need to remember different IPs or domains for every single app
🔐
SSL certs must be managed per LB — HTTPS setup repeated for each service. Nightmare to maintain.
⚙️
No centralized routing rules — path-based or domain-based routing is impossible with plain LoadBalancers
Solution: Ingress — one entry point, smart routing rules, one IP for all your apps
❌ Without Ingress — Chaos!

Definition
What is Ingress?
🌐
Kubernetes object for external HTTP/HTTPS traffic — manages how the outside world reaches your apps
🎯
Routes traffic to correct Services — using URL paths, domains, or headers to decide where each request goes
🧠
Smart traffic manager — like an intelligent receptionist who reads the request and directs it to the right team
☝️
One entry point for all apps — one IP, one domain, multiple applications. Clean and cost-effective.
Important: Ingress is just the rules object. You need an Ingress Controller (e.g. nginx, traefik) to actually execute those rules.

04 / 09
Features
Key Features of Kubernetes Ingress
📍
Path-Based Routing
Route by URL path
One domain, different paths go to different services based on the URL path after the domain.
example.com/api → api-svc
example.com/web → web-svc
🏷
Host-Based Routing
Route by subdomain
Different subdomains or hostnames route to completely different services in the cluster.
api.example.com → api-svc
app.example.com → web-svc
🔐
TLS/SSL Termination
HTTPS at the edge
Handle HTTPS at the Ingress layer. Backend services can stay plain HTTP — Ingress decrypts for them.
HTTPS → Ingress → HTTP → Pods
Certificate managed in one place
🎛
Centralized Management
All traffic in one place
Define all your routing rules in one YAML file. Easy to review, update, and version control.
One Ingress YAML rules everything
GitOps friendly, auditable

Feature Deep Dive
Path-Based Routing Explained
📍
Same domain, different paths — one domain (example.com) serves multiple apps based on URL path
🔀
Ingress matches the path prefix — checks if request starts with /api, /web, or /auth
🎯
Forwards to the matching Service — path prefix determines which backend Service receives the traffic
🌍
User sees one clean URL — all hidden behind example.com. Microservices stay transparent.
# Ingress YAML rules
- path: /api  → backend: api-service:8080
- path: /web  → backend: web-service:80
- path: /auth → backend: auth-service:3000

Feature Deep Dive
Host-Based Routing Explained
🏷
Different subdomains, different apps — api.example.com and app.example.com are two different apps
🌐
Ingress reads the Host header — every HTTP request carries a Host header. Ingress matches it to its rules.
🏢
Perfect for multi-tenant systems — each team has their own subdomain, each routed to their own Service
🔗
Can combine with path routing — api.example.com/v1 and api.example.com/v2 for versioning
# Ingress YAML — host routing
- host: api.example.com → api-service
- host: app.example.com → web-service
- host: admin.example.com → admin-service
