The Problem
Why Hardcoding Config is Dangerous
🔒
Config hardcoded inside the image — database URLs, API keys, ports baked into the container. Inflexible by design.
🔄
Every config change = rebuild container — updating a single setting like a port number forces a full image rebuild and redeploy
🌍
Different environments, different configs — dev uses localhost DB, prod uses cloud DB. Same image cannot serve both without modification.
🐛
Easy to introduce errors — wrong env variable committed to code, pushed to production. Debugging is painful and slow.
🔐
Sensitive data exposed in image — hardcoded credentials inside a Docker image are visible to anyone who pulls it
Solution needed: Store config outside the application. Inject it at runtime. Change config without touching the container image. That's ConfigMap.
❌ Hardcoded Config — Inside Image
// hardcoded in app.js
const DB_URL = "postgres://prod";
const PORT = 8080;
🔄 Change DB → rebuild image
🌍 Can't reuse across envs
🔐 Secrets visible in image
✅ ConfigMap — External Config
⚙ ConfigMap: DB_URL, PORT
↓ inject at runtime
📦 Pod reads config automatically
✅ Change config → no rebuild
02 / 09
Definition
What is a ConfigMap?
⚙
Kubernetes object that stores non-sensitive config data — key-value pairs, config files, environment variables — all in one place
🔓
Separates configuration from application code — your app image stays the same. Only the ConfigMap changes per environment.
📦
Used by Pods and containers — injected as environment variables, mounted as files, or used as command-line arguments
🌿
Makes applications more flexible — deploy the same container image to dev, test, and prod — only ConfigMap differs between them
🗂
Centralized configuration management — all config lives in one K8s object. Easy to update, version, and audit via kubectl.
Important: ConfigMap is for non-sensitive data only. Passwords and tokens go in Secrets (next topic).
ConfigMap YAML
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_URL: "postgres://prod-db"
  LOG_LEVEL: "info"
  PORT: "8080"
  ENV: "production"

Injection Flow
How ConfigMap Works — Step by Step
1
Admin/Dev creates ConfigMap
ConfigMap YAML written with key-value config data. Applied to cluster with kubectl apply.
⚙
↓
2
ConfigMap stored in Kubernetes cluster
K8s stores the ConfigMap in etcd. It's now available as a named resource in the namespace.
🗂
↓
3
Pod spec references ConfigMap
Pod YAML references ConfigMap by name under envFrom or volumes. Kubernetes handles injection.
📦
↓
4
Config injected into container at startup
Kubernetes injects ConfigMap values as env vars or mounts config file at the declared path.
💉
↓
5
App reads config dynamically
App reads process.env.DB_URL or opens /config/app.conf — config is live. No hardcoding. ✅

Usage Methods
3 Ways to Use ConfigMap in a Pod
🌿
Method 1 — Environment Variables
Most common · Simplest to use
⭐ Popular
# Pod YAML
envFrom:
  - configMapRef:
    name: app-config
# Result in container:
$DB_URL = postgres://prod
$PORT = 8080
All key-values become env vars ✅
🎯
Method 2 — Specific Keys
Selective injection
# Pod YAML — pick specific keys
env:
  - name: DATABASE_URL
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: DB_URL
Pick only the keys you need ✅
📁
Method 3 — Mounted Files
Full config file injection
# Mounted as a file at path
volumes:
  - name: config-vol
    configMap:
      name: app-config
# Result: /etc/config/nginx.conf
Config file available at mount path ✅

Benefits
Why Use ConfigMaps?
🔓
Decouples config from code — application image stays the same. Only the ConfigMap changes between environments. Clean separation of concerns.
🌍
Easier environment management — one ConfigMap per environment (dev, test, prod). Same container image runs everywhere.
⚡
Faster config updates — update ConfigMap with kubectl. No image rebuild, no long CI/CD pipeline. Changes apply in seconds.
🚀
Better portability — move application to a new cluster? Just bring the ConfigMap. App automatically picks up all settings.
📋
Simplifies deployments — no special scripts to manage environment variables. Kubernetes injects config automatically at Pod startup.
Key principle: Build once, configure many times. Same Docker image, different ConfigMaps per environment — ideal 12-factor app pattern.

🏭

🔄
