Overview
Control Plane — All 4 Components
🔌
API Server
kube-apiserver
The front door — every request from kubectl, UI, or any tool enters here first. Validates, processes, and routes everything.
Entry Point for ALL requests
📅
Scheduler
kube-scheduler
The decision maker — watches for new Pods with no node assigned, then picks the best node based on resources and rules.
Assigns Pods → best Node
🔄
Controller Manager
kube-controller-manager
The maintainer — runs many controllers in one process. Constantly watches cluster state and fixes any difference from desired.
Detects & Fixes drift
🗄
etcd
key-value store
The memory — stores all cluster data: config, state, secrets. Every change is saved here. If etcd is healthy, the cluster is recoverable.
Source of Truth
🔑 Key rule: All components communicate exclusively through the API Server — no direct component-to-component calls

Component 2
Scheduler — The Decision Maker
👀
Watches for unscheduled Pods — as soon as a new Pod is created, Scheduler sees it
🖥
Checks resource availability — scans all nodes for CPU, memory, and storage space
📋
Respects constraints & rules — node selectors, affinity rules, taints & tolerations
✅
Assigns Pod to best Node — tells API Server the decision, which then updates etcd
Note: Scheduler only decides where to run — the Kubelet on the worker node actually runs it

Component 3
Controller Manager — The Maintainer
🔁
Runs a continuous control loop — forever checks: is actual state = desired state?
🔍
Detects drift — if desired = 3 Pods but only 2 are running, it notices immediately
🔧
Automatically fixes issues — creates a new Pod to replace the failed one

06 / 09
Component 4
etcd — The Data Store
💾
Stores all cluster data — every object, config, and state lives here as key-value pairs
📖
Key-value database — simple concept: key = "pod/name", value = all pod details in JSON
🔐
Stores secrets & config — ConfigMaps, Secrets, RBAC rules all stored securely
🔒
Only API Server reads/writes — no other component touches etcd directly
⚡
Distributed & consistent — uses Raft consensus algorithm to stay reliable
⚠ Critical: If etcd data is lost and not backed up, the entire cluster configuration is gone
