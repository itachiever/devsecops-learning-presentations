The Problem
Why Can't Pods Use Storage Directly?
🔧
Pods should not manage storage details — knowing disk types, server IPs, or cloud credentials violates separation of concerns
👨‍💻
Developers shouldn't worry about infrastructure — a dev writing a DB app should not need to configure an AWS EBS disk
🔗
Tight coupling creates problems — if storage config is hardcoded in Pod spec, moving to a new cluster means rewriting everything
📦
Pods are replaced frequently — new Pods need a clean, simple way to reconnect to the same storage automatically
Solution needed: An abstraction layer where apps simply request storage without worrying about how or where it's provided. That's PVC.

Definition
What is a Persistent Volume Claim?
📋
A formal request for storage — PVC is a Kubernetes object where an application says "I need this much storage with these properties"
👤
Created by the developer/user — you write a YAML with size and access mode. Kubernetes does the matching for you.
🔗
Kubernetes finds a matching PV — K8s automatically scans available PVs and binds the best match to your PVC
🌉
Acts as a bridge between Pod and PV — Pod never talks to PV directly. PVC is always the intermediary layer.
♻️
Simplifies storage across environments — same PVC YAML works on AWS, GCP, Azure — backend changes transparently
Simple analogy: PVC = filling a storage request form 📝. You write what you need, hand it to Kubernetes, and K8s finds you the right storage room.
PVC — What It Contains

kind: PersistentVolumeClaim

How PVC Works — From Request to Storage

Admin provisions a Persistent Volume (PV)
A 100GB disk is created (AWS EBS / GCP / NFS). PV is Available in the cluster waiting to be claimed.
👨‍💼
↓
2
Developer creates a PVC
PVC YAML specifies: storage size (10Gi) + access mode (ReadWriteOnce). Applied to cluster.
📋
↓
3
Kubernetes matches and binds PVC → PV
K8s finds a PV that satisfies size + access mode requirements. Status: PVC = Bound, PV = Bound.
🔗
↓
4
Pod mounts the PVC and accesses storage
Pod spec references the PVC by name. K8s mounts the bound PV into the Pod at /data path.
📦
↓
5

New Pod re-mounts same PVC → same PV → all data still there. No data loss, no re-configuration.

Architecture
Pod → PVC → PV → Storage — Layered Architecture
📦
Layer 1 — Pod
Application container. Needs storage to read/write data. Mounts PVC by name in Pod spec.
App layer
↕ uses PVC by name
📋
Layer 2 — PVC (Abstraction)
Storage request object. Specifies size + access mode. Hides all infrastructure details from the Pod.
⭐ Bridge layer
↕ K8s binds to matching PV
🗄
Layer 3 — PV (Cluster Resource)
Persistent Volume provisioned by admin. Knows the real storage backend. Matched to PVC by Kubernetes.
K8s resource
↕ backed by real disk
☁️
Layer 4 — Actual Storage Backend
Real physical or cloud storage. AWS EBS · GCP Disk · NFS · Azure Disk. App never touches this directly.

Capabilities
Key Features of Persistent Volume Claims
🎯
Simplifies storage management — developers only specify what they need (size, access mode). Kubernetes handles the rest automatically.
⚡
Dynamic storage allocation — with StorageClass, PVs are auto-created on demand. No manual PV provisioning required.
🔓
Decouples app from infrastructure — PVC is the abstraction layer. Same app works on any cloud or on-prem storage backend.
🛡
Supports persistent data storage — PVC stays bound even if Pod is deleted. New Pods reconnect to the same data via same PVC.
♻️
Flexible and reusable — PVC can be reused across different Pods. Multiple Pods can share one PVC (with RWX access mode).
Developer experience: Without PVC you manage disks. With PVC you just describe what you need — Kubernetes becomes your storage manager.

Access Modes
PVC Access Modes — How Storage is Shared
1️⃣
ReadWriteOnce
RWO ⭐ Most common
Volume can be mounted as read-write by one single node at a time. Perfect for databases — MySQL, Postgres.
📦 Pod A → ✏️ read+write
📦 Pod B → ✗ blocked
2️⃣
ReadOnlyMany
ROX
Volume can be mounted as read-only by many nodes simultaneously. Good for shared config files or static assets.
📦 Pod A → 👁 read only
📦 Pod B → 👁 read only
3️⃣
ReadWriteMany
RWX
Volume can be mounted as read-write by many nodes simultaneously. For shared file systems like NFS.
📦 Pod A → ✏️ write
📦 Pod B → ✏️ write
Access Mode Summary
RWO
1 node · read + write
✅ Databases (MySQL, PG)
ROX
Many nodes · read only
✅ Config / static assets
RWX
Many nodes · read + write
✅ NFS / shared files
Specified in PVC YAML
accessModes field
