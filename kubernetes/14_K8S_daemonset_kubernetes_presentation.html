The Problem
Some Apps Must Run on Every Node
📋
Log agents need every node covered — if a node has no log agent, its container logs are completely lost
📊
Monitoring agents need every node — missing a node means blind spots in your cluster metrics
🌐
Network plugins run per-node — CNI plugins must exist on every node to enable pod networking
🎲
Deployment can't guarantee this — Deployment spreads Pods randomly. A node could have 2 Pods or 0 Pods.
Solution: DaemonSet guarantees exactly one Pod on every node — always, automatically

Definition
What is a DaemonSet?
🌿
Ensures one Pod runs on EACH node — not one or two, but exactly one per node, every time
➕
Auto-adds Pods to new nodes — new node joins the cluster? DaemonSet immediately schedules a Pod on it
➖
Auto-removes Pods when nodes leave — node removed? Its DaemonSet Pod is cleaned up automatically
🎯
Designed for system-level services — background agents that need cluster-wide presence, not application Pods
Key rule: N nodes in your cluster = N Pods created by DaemonSet. Always. Automatically.

Behavior
Important DaemonSet Behaviors
☑
Exactly one Pod per node by default — the core guarantee. Never more, never less on each node
🏷
Can target specific nodes with selectors — only run on GPU nodes, only on SSD nodes, only on specific zones
⚙️
Tolerates system taints — DaemonSet Pods can run on master/control-plane nodes that regular Pods cannot
🔄
Pod restarts follow normal policy — if the DaemonSet Pod crashes, Kubelet on that node restarts it automatically
Node selector example in YAML
nodeSelector:
  disktype: ssd  # only run on SSD nodes
