The Problem
Why Resource Management Matters
🤝
Multiple Pods share the same node resources — CPU, memory, and disk are finite. Without rules, workloads compete aggressively with each other.
💣
One greedy app starves others — a memory leak or CPU spike in one Pod can exhaust node resources and cause other Pods to crash or become unresponsive
🎯
Scheduler needs resource hints — without requests, the Kubernetes scheduler cannot make intelligent placement decisions. Pods may land on overloaded nodes.
⚖
Resource requests = fair allocation — each Pod declares what it needs. Scheduler places it on a node with enough capacity. No starvation.
🔒
Resource limits = protection boundaries — prevents any single container from consuming unbounded resources. Cluster stability guaranteed.
Without limits: One buggy app with a memory leak can OOMKill every other Pod on the node. The whole node goes down. Resources = first-class concern.

Resource Requests
What are Resource Requests?
📋
Minimum resources a container needs — "I need at least 200m CPU and 256Mi memory to run properly." Kubernetes guarantees this much.
🗺
Used by the scheduler for Pod placement — Kubernetes finds a node with enough free capacity to satisfy the request. Pod only lands where it fits.
🛡
Guaranteed allocation on the node — the requested CPU and memory are reserved for the container. Other Pods cannot take them away.
🔍
Scheduler compares requests vs available — a node with 4 CPU and 3 already-requested will only accept new Pods requesting ≤1 CPU
✅
Not a hard cap — containers CAN use more than the request if resources are available on the node. The request is the floor, not the ceiling.
Analogy: Request = reserved seat on a flight ✈. Kubernetes guarantees you have a place. You won't be bumped off even if the plane is full.
Request in Pod YAML
resources:
  requests:
    cpu: "200m"
    memory: "256Mi"
☸ Scheduler Decision
📦 Pod needs: 200m CPU, 256Mi mem
🖥 Node A: 150m free → ❌ skip
🖥 Node B: 800m free → ✅ schedule here
200m CPU guaranteed on Node B ✅
03 / 10
Resource Limits
What are Resource Limits?
🔒
Maximum resources a container can consume — "You can use at most 500m CPU and 512Mi memory." Hard ceiling enforced by the kernel.
🛡
Protects the node and other Pods — even if an app has a memory leak or CPU bug, it cannot consume resources beyond the limit. Others stay safe.
⚡
CPU is throttled at the limit — if a container tries to use more CPU than its limit, the kernel throttles it. Performance drops but it keeps running.
💀
Memory limit violations = OOMKill — if a container exceeds its memory limit, the kernel sends SIGKILL immediately. No warning, instant termination.
⚖
Limit ≥ Request always — the limit must be equal to or greater than the request. Setting limit lower than request is an error.
Analogy: Limit = luggage weight allowance on a flight ✈. You can bring up to 23kg. Try to bring more — the airline will refuse (OOMKill) or charge you (throttle).
Limit in Pod YAML
resources:
  requests:
    cpu: "200m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
⚡ CPU > 500m → kernel throttles
💀 Memory > 512Mi → OOMKill!
✅ Other Pods stay protected

Resource Units
CPU & Memory — How They're Measured
⚡
CPU
Measured in cores or millicores
1 CPU = 1000 millicores (m)
1 full CPU core = 1000m. So 500m = half a core, 250m = quarter core.
100m
light app
500m
medium app
2000m / 2
heavy app
⚡ CPU is compressible — throttled, not killed
🧠
Memory
Measured in MiB, GiB, or bytes
Memory units
Mi = Mebibyte (1024 KiB), Gi = Gibibyte. Use Mi or Gi, not MB or GB.
128Mi
light svc
512Mi
web app
2Gi
database
💀 Memory is incompressible — exceeding limit = OOMKill
QoS Classes
⭐ Guaranteed
Request = Limit for both CPU & mem. Highest priority, never evicted.
🔶 Burstable
Request < Limit. Can burst. Evicted if node is under pressure.
⚠ BestEffort
No requests or limits set. Evicted first under pressure.

Limit Enforcement
What Happens When Limits are Exceeded?
⚡
CPU Throttling — when a container tries to use more CPU than its limit, the Linux CFS scheduler throttles it. App runs slower but stays alive. No crash.
💀
Memory OOMKill — when a container exceeds its memory limit, the Linux OOM killer sends SIGKILL immediately. Pod shows OOMKilled in status.
🔄
Pod may restart (CrashLoopBackOff) — if a Pod keeps OOMKilling on startup, Kubernetes enters CrashLoopBackOff with exponential backoff delays
📊
Monitor kubectl describe pod — kubectl describe pod <name> shows OOMKilled reason and exit code 137 (killed by signal 9)
✅
Fix by increasing limits or optimizing app — check actual usage with kubectl top pods, then adjust limits based on real data
Key difference: CPU over-limit → throttled (slow but running). Memory over-limit → killed immediately (OOMKilled). Memory is unforgiving.

💀 Memory — OOMKilled (instant)

CPU Limit 500m: During traffic spikes, the app can burst up to 500m. Exceeding 500m causes throttling — degraded but alive.

Best Practices
Resource Requests & Limits — Best Practices
📋
Always Set Both Request & Limit
Never leave them blank. No requests = poor scheduling. No limits = BestEffort QoS, evicted first under pressure.
📊
Monitor Real Usage First
Use kubectl top pods or Prometheus. Set requests near p50 usage, limits near p99. Don't guess.
⚖
Don't Set Limits Too Tight
Overly tight memory limits cause constant OOMKills. Allow at least 50-100% headroom above typical usage for burst capacity.
🏭
Use LimitRange for Defaults
Create a LimitRange object in each namespace to set default requests/limits. Pods without specs inherit these.
🚫
Avoid Extremely High Limits
Setting limits.cpu: "100" won't help. It just reduces scheduling efficiency and allows node exhaustion.
🔍
Use VPA for Auto-Tuning
Vertical Pod Autoscaler (VPA) analyzes usage patterns and automatically recommends or updates requests/limits. Great for production.
