The Brain
Control Plane
The Control Plane is the decision-making layer of Kubernetes. It manages the entire cluster from one place.
🌐
Makes global decisions — decides what runs, where it runs, and when to restart
📊
Manages cluster state — always knows what's running and what should be running
🔄
Handles scheduling — assigns containers to the best available worker node
🛡
Maintains desired state — if something breaks, it automatically fixes it
🧠

etcd

04 / 09
Deep Dive — Control Plane
Control Plane Components Explained
🔌
API Server
kube-apiserver
The front door to the cluster — every request (from kubectl, UI, or any tool) comes here first. Validates and processes all API calls.
📅
Scheduler
kube-scheduler
Decides which worker node a new Pod should run on — checks available CPU, memory, and constraints before assigning.
🔄
Controller Manager
kube-controller-manager
Runs multiple controllers in one process — Node Controller, Replication Controller, Job Controller. Ensures desired state is always maintained.
🗄
etcd
key-value store
The cluster's memory — stores all configuration data, current state, and secrets. If etcd is healthy, the cluster is recoverable.

Deep Dive — Worker Node
Worker Node Components Explained
🤝
Kubelet
Node Agent
The main agent running on every worker node. Receives Pod specs from the API Server, ensures containers are running correctly, and reports status back to the Control Plane.
🐳
Container Runtime
containerd / CRI-O
The software that actually pulls images and runs containers. Kubernetes uses the CRI (Container Runtime Interface) — so it works with Docker, containerd, or CRI-O.
🌐
kube-proxy
Network Proxy
Handles all networking rules on each node. Routes traffic from Services to the correct Pods — acts as the local network router for the node.
📦
Pod
Smallest K8s unit
A Pod is a wrapper around one or more containers. Containers in the same Pod share network and storage. Pods are created, managed, and deleted by Kubernetes automatically.
