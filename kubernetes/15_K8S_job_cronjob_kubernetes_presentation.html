The Problem
Not All Workloads Run Forever
🔁
Deployment runs Pods continuously — great for web servers, but what about one-time database backup?
💾
Backup jobs need to run & stop — run once at midnight, complete, then exit. Not keep running.
📊
Data processing has a clear end — process a file, generate a report. Done. No need to keep the Pod alive.
🎲
Deployment would restart completed Pods — Deployment thinks completion = failure, keeps restarting it forever!
Solution: Job for one-time tasks · CronJob for scheduled recurring tasks

What is a Job?
▶
Runs a task until completion — creates one or more Pods to run the task and tracks whether they finish
✅
Ensures task runs successfully — if a Pod fails, Job creates a new one. Keeps trying until success.
🔁
Retries on failure — you define backoffLimit: how many times to retry before giving up
🛑
Stops after completion — once all Pods complete successfully, Job marks itself as done and stops
Key difference: Unlike Deployment, a Job's Pod completing successfully is the desired state — not a failure to restart

Pod kept for log inspection. Job history retained.
03 / 09
Job Features
Key Features of a Job
1️⃣
One-time execution — run a task once and stop. Perfect for migrations, reports, data imports
🔁
Retry on failure — backoffLimit: 3 means try up to 3 times before declaring failure
⚡
Parallel execution — set parallelism: 4 to run 4 Pods at once and process faster
🎯
Completions count — set completions: 10 to process 10 tasks in total
# Job YAML example
apiVersion: batch/v1
kind: Job
spec:
  completions: 1  # run once
  backoffLimit: 3  # retry 3 times
  parallelism: 1  # 1 pod at a time

What is a CronJob?
⏰
Runs Jobs on a defined schedule — specify when to run using a cron expression (e.g., every midnight)
🐧
Similar to Linux cron — if you know Linux's crontab, you already understand CronJob
🔄
Automates repetitive tasks — set it once, runs automatically forever at the scheduled time
⚡
Creates a new Job each time — every schedule trigger creates a fresh Job and Pod internally
Cron syntax: "0 2 * * *" = run at 2:00 AM every day

Relationship
CronJob → Job → Pod
CronJob creates a Job on schedule. Job creates a Pod to do the work.
✅
Job completes & stops
Success = Pod exits 0. Deployment restarts it — Job celebrates it.
⏰
CronJob repeats forever
Set schedule once, automatic execution forever
🔁
Retry on failure
backoffLimit controls how many retries before failing
⚡
batch/v1 API group
Both use apiVersion: batch/v1
"Job for when you need it done once. CronJob for when you need it done on a schedule — automation at its best."
