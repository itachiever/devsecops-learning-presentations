Why It Matters
Why Kubernetes Security Cannot Be Ignored
🏭
Kubernetes runs critical production systems — databases, payment APIs, user data — a breach doesn't just take down a server. It can take down the whole business.
👥
Multiple users share cluster resources — developers, operators, CI/CD pipelines all access the same cluster. Without controls, anyone can do anything.
💀
One breach can compromise everything — a vulnerable container can be used to attack other Pods, access Secrets, or take over the entire cluster
⚙
Misconfigurations are the #1 risk — most K8s breaches happen due to wrong RBAC, open ports, or exposed dashboards — not advanced hacks
🏗
Security must be built-in, not bolted on — adding security after deployment is expensive and incomplete. Secure every layer from the start.
Shift left: Security must be considered at design time, not added after production. Every Kubernetes practitioner must understand security basics.
💀 Unsecured Cluster
❌ Anyone can create/delete Pods
❌ Containers run as root
❌ Secrets visible to all
❌ All Pods talk to each other
❌ No image scanning
🛡 Secured Cluster
✅ RBAC — least privilege access
✅ Non-root containers
✅ Secrets encrypted + RBAC
✅ Network Policies enforced
✅ Images scanned & verified

Access Control
Authentication & Authorization
👤
Authentication = "Who are you?" — verifies identity. Kubernetes checks certificates, tokens, or OIDC before allowing any request
🎯
Authorization = "What can you do?" — after identity is confirmed, K8s checks if that user/service account has permission for the action
⚙
RBAC — Role-Based Access Control — the standard K8s authorization system. Defines Roles (what) and RoleBindings (who gets that role)
🤖
Service Accounts for workloads — Pods get their own identity via ServiceAccount. Used by apps to call the Kubernetes API securely
⚠
Least privilege is critical — grant only the minimum permissions needed. A compromised Pod with admin rights = full cluster compromise
RBAC rule: Never use ClusterAdmin for regular workloads. Create specific Roles with only the permissions the app actually needs.

Container Hardening
Pod & Container Security
🚫
Never run containers as root — a root container that gets compromised gives the attacker root access to the host node. Always use a non-root user.
🔒
Use Security Contexts — define runAsUser, readOnlyRootFilesystem, and allowPrivilegeEscalation: false
⚡
Drop unnecessary Linux capabilities — containers inherit host capabilities by default. Drop all and add only what's needed.
📋
Pod Security Admission (PSA) — Kubernetes built-in policy engine. Enforces Baseline or Restricted profiles across namespaces.
🔒
Read-only root filesystem — prevent attackers from writing malicious files to the container if it is compromised
Rule: Least privilege for containers = run as non-root, read-only FS, no extra capabilities. Default = deny all extras.
❌ Insecure Container
securityContext:
  runAsRoot: true ❌
  privileged: true ❌
  allowPrivEscalation: true ❌
✅ Secure Container
securityContext:
  runAsUser: 1000 ✅
  runAsNonRoot: true ✅
  readOnlyRootFS: true ✅
  allowPrivEscalation: false ✅
  capabilities: drop: [ALL] ✅

Network Isolation
Network Security & Policies
🌐
By default all Pods can talk to all Pods — a K8s cluster is fully open by default. Every Pod can reach every other Pod. This is dangerous in production.
📋
Network Policies restrict traffic — define exactly which Pods can communicate with which other Pods using label selectors and rules
🔒
Default deny strategy — block all traffic first, then explicitly allow only what's needed. A frontend Pod should NOT reach the database directly.
🔐
Use TLS for service communication — encrypt traffic between services using mutual TLS (mTLS). Service meshes like Istio can automate this.
🌍
Control ingress and egress — restrict what comes into and what goes out of your cluster. Prevent data exfiltration and unauthorized access.
Best practice: Apply a default-deny NetworkPolicy to every namespace. Then explicitly allow only the connections your app actually needs.

Sensitive Data
Secrets & Sensitive Data Protection
🔐
Never hardcode credentials — passwords, tokens, keys in source code get committed to Git and shared. One leak = full system compromise.
📦
Use Kubernetes Secrets — store sensitive data base64-encoded, access-controlled via RBAC, and mountable securely into Pods
🛡
Enable encryption at rest — by default etcd stores Secrets unencrypted. Enable etcd encryption to protect data if the disk is compromised.
🏛
Use external secret stores — for production, integrate HashiCorp Vault or AWS Secrets Manager. Secrets are never stored in etcd directly.
🔄
Rotate secrets regularly — change passwords and tokens on a schedule. Kubernetes Secrets make rotation easy — update and restart Pod.
Vault analogy: Don't leave the password on a sticky note (hardcoded). Put it in a locked vault (Secret) and give only the right people the key (RBAC).
Secret Security Levels
❌ Level 0 — Worst
Hardcoded in source code / Docker image
⚠ Level 1 — Bad
Stored in ConfigMap (plain text, visible)
✅ Level 2 — Good
Kubernetes Secret (RBAC + base64)
🏆 Level 3 — Best
K8s Secret + etcd encryption + Vault

Supply Chain
Image & Supply Chain Security
🐳
Use trusted base images only — start from official images (nginx:alpine, python:3.12-slim). Never use random images from unknown authors on Docker Hub.
🔍
Scan images for vulnerabilities — use tools like Trivy, Snyk, or Grype. Scan every image in your CI/CD pipeline before it goes to production.
🔄
Keep images up to date — old images have old CVEs (Common Vulnerabilities). Rebuild images regularly to get the latest patched packages.
✍
Sign and verify images — use Cosign + Sigstore to sign images and enforce that only signed images run in your cluster
🏠
Use a private registry — push verified images to your own registry (ECR, GCR, ACR, Harbor). Restrict access to prevent unauthorized image pulls.
Pipeline rule: Build → Scan → Sign → Push → Deploy. Never deploy an image that hasn't passed a vulnerability scan.

Security Checklist
Kubernetes Security Best Practices
🎯
Least Privilege Everywhere
RBAC roles with minimum permissions. Non-root containers. Drop all Linux capabilities. No wildcard permissions.
🔄
Keep Everything Updated
Upgrade Kubernetes regularly. Patch node OS. Rebuild container images to get latest security fixes. Update dependencies.
🌐
Default-Deny Networking
Apply NetworkPolicies blocking all traffic by default. Explicitly allow only required connections. Use TLS between services.
🔍
Scan Images in CI/CD
Block deployment of images with HIGH/CRITICAL CVEs. Use Trivy or Snyk. Sign images with Cosign. Use admission controllers.
🔐
Protect Secrets Properly
Enable etcd encryption. Use Vault for production. Rotate credentials regularly. Never store secrets in ConfigMaps or code.
📊
Monitor & Audit Everything
Enable audit logging. Use Falco for runtime threat detection. Monitor RBAC changes. Set up alerts for anomalous activity.
