The Problem
Why Kubernetes Needs RBAC
👥
Multiple teams share one cluster — developers, testers, CI/CD pipelines, ops teams all work in the same Kubernetes cluster simultaneously
💀
Without RBAC everyone has full access — any developer could delete production databases, modify any namespace, or read all Secrets cluster-wide
🎯
Different teams need different permissions — dev team reads logs, ops team restarts Pods, CI/CD creates Deployments, but no one needs admin access
⚠
Accidents and attacks both require access — a bug in CI/CD code or a compromised token can delete critical resources if access isn't restricted
🛡
RBAC enforces least privilege — each user and service gets only the minimum permissions needed to do their job. Nothing more.
Real risk: A compromised developer token with admin access = full cluster takeover. RBAC limits the blast radius of any breach.
❌ Without RBAC — Everyone is Admin
🧑‍💻 Dev → can delete prod DB
🤖 CI/CD → can read all Secrets
🐛 Bug → wipes entire namespace
🔓 Intern → cluster admin access
✅ With RBAC — Least Privilege
🧑‍💻 Dev → get/list pods only
🤖 CI/CD → create deployments only
👮 Ops → restart pods, view logs
👑 Admin → full access (admin only)

Definition
What is RBAC?
📖
RBAC = Role-Based Access Control — a security mechanism that controls who can perform which actions on which Kubernetes resources
🎭
Permissions are assigned to Roles, not users directly — you create a Role with permissions, then assign that Role to users via RoleBindings
🗺
Three core concepts: Role (what actions), RoleBinding (who gets the role), Subject (user/group/service account)
🌐
Works at namespace or cluster level — Roles are namespace-scoped, ClusterRoles apply across the entire cluster
✅
Built into Kubernetes — enabled by default — RBAC is the standard authorization mode for all modern Kubernetes clusters
Analogy: RBAC = job roles in a company 🏢. A "Junior Dev" role has defined responsibilities. Assigning Alice to that role gives her exactly those permissions.

Role Object
What is a Role?
📋
Role defines a set of allowed actions — it specifies which verbs (get, list, create, delete) are allowed on which resources (pods, secrets, deployments)
🗺
Namespace-scoped — a Role only applies within a specific namespace. A Role in "dev" namespace doesn't affect "prod" namespace.
⚡
Verbs = allowed actions — get list watch create update delete
🎯
Resources = what the role controls — pods, deployments, services, secrets, configmaps, nodes, namespaces, and more
⚠
Role alone does nothing — a Role just defines permissions. Without a RoleBinding, no user has that Role. They must be linked.
Analogy: A Role = a job description 📄. It lists what a person in that job is allowed to do — but doesn't assign anyone to it yet.
Role YAML Example
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get","list","watch"]
✅ Can: GET a specific pod
✅ Can: LIST all pods in namespace
❌ Cannot: DELETE pods
❌ Cannot: access other namespaces

RoleBinding Object
What is a RoleBinding?
🔗
RoleBinding links a user to a Role — it's the bridge that says "Alice gets the pod-reader Role in the dev namespace"
👤
Subject = who receives the permissions — can be a User (human), Group (team), or ServiceAccount (application/bot)
📋
References an existing Role by name — the RoleBinding just points to the Role. Change the Role → all bound users automatically get updated permissions
🗺
Also namespace-scoped — a RoleBinding in the "dev" namespace grants permissions only within "dev". It cannot grant access to "prod".
♻️
One Role, many RoleBindings — create one "developer" Role, then bind multiple users to it. No need to duplicate permission rules.
Analogy: RoleBinding = HR letter assigning Alice to the "Developer" job role 📄. The role exists, now Alice officially has it.
RoleBinding YAML
kind: RoleBinding
metadata:
  name: read-pods-alice
  namespace: dev
subjects:
- kind: User
  name: alice
roleRef:
  kind: Role
  name: pod-reader
👤 alice
→
🔗 RoleBinding
→
📋 Role
✅ alice can GET/LIST pods in dev

Cluster-Wide Scope
ClusterRole & ClusterRoleBinding
📋
Role + RoleBinding
Namespace-scoped
🗺 Namespace
Permissions apply only within a specific namespace. A Role in "dev" cannot access "prod". Ideal for team or environment isolation.
kind: Role
namespace: dev # only in dev
✅ Use for team namespaces
✅ Isolate dev/test/prod
✅ Most common for developers
vs
🌐
ClusterRole + ClusterRoleBinding
Cluster-wide scope
🌐 All namespaces
Permissions apply across all namespaces or cluster-level resources (nodes, namespaces, PVs). Use for admins and global operations.
kind: ClusterRole
# no namespace field = cluster-wide
✅ Cluster admins
✅ Node and PV management
⚠ Use sparingly — powerful
Quick Reference
📋 Role
Permissions in 1 namespace
🔗 RoleBinding
Assigns Role in 1 namespace
🌐 ClusterRole
Permissions cluster-wide
🌐 ClusterRoleBinding
Assigns ClusterRole globally

Real-World Example
Developer RBAC in Action
🧑‍💻
Scenario: Alice is a backend developer who needs access to the "api" namespace to debug and monitor her application
📋
Role created: "dev-role" allows get/list/watch on pods, logs, and deployments — nothing destructive
🔗
RoleBinding created: "alice-dev-binding" links alice to "dev-role" in the "api" namespace only
✅
Alice can: view pods, check logs, see deployments — everything needed to debug. Fast and effective.
❌
Alice cannot: delete pods, access secrets, touch other namespaces, or escalate her own privileges
Result: Alice can do her job effectively. Even if her credentials are compromised, the attacker can only read pods in "api" namespace. Blast radius contained.
Alice's Access Dashboard
👤 alice — namespace: api
✅ kubectl get pods -n api
✅ kubectl logs <pod> -n api
✅ kubectl get deployments -n api
❌ kubectl delete pod -n api → 403
❌ kubectl get secrets -n api → 403
❌ kubectl get pods -n prod → 403

Best Practices
RBAC Security Best Practices
🎯
Always Least Privilege
Grant only the permissions needed for the job. Start with minimal access and add more only if required. Never start with admin and restrict later.
🚫
Avoid Wildcard Permissions
verbs: ["*"] and resources: ["*"] are dangerous anti-patterns. Always specify exactly what's needed.
👥
Bind Groups, Not Users
Bind RoleBindings to Groups instead of individual users. Add/remove users from the group. Easier to maintain at scale.
🤖
Use ServiceAccounts for Apps
Never use human user credentials for applications. Create dedicated ServiceAccounts for each workload with minimal permissions.
🔍
Audit RBAC Regularly
Run kubectl auth can-i --list to review permissions. Remove stale RoleBindings for departed employees.
📏
Prefer Role over ClusterRole
Use namespace-scoped Roles by default. Only escalate to ClusterRole when truly needed (admins, monitoring agents, node-level access).
