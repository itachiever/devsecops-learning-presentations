The Problem
Life Without Deployment
🧑‍💻
Pods managed manually — create, update, delete one by one. Error-prone and slow.
⚙️
ReplicaSet can't do updates — it only maintains count. Changing the app image? You're on your own.
⬇️
Downtime during updates — stopping old Pods then starting new ones means users see errors
↩️
No easy rollback — if a bad version is deployed, reverting it is complex and risky
💡 Solution: Deployment handles all of this — updates, rollbacks, scaling — all automated

Definition
What is a Deployment?
🏆
Higher-level Kubernetes object — sits above ReplicaSet in the hierarchy. The recommended way to run apps.
⚙️
Manages ReplicaSets and Pods — creates and controls ReplicaSets, which in turn manage Pods
🔄
Used to deploy and update apps — change the image version? Deployment handles the transition safely
🎯
Maintains desired state — always keeps your app at the right version with the right number of replicas
In practice: Every production app in Kubernetes uses a Deployment — it's the standard and recommended approach
Deployment controls everything

Key Feature
Rolling Update — Zero Downtime
🔄
Update app without stopping it — users keep using the app while you deploy the new version
📈
Old Pods replaced gradually — K8s creates a new Pod, waits for it to be ready, then removes an old one
✅
Continuous availability guaranteed — at no point are all Pods down. Always some running.
🎮
Fully controlled update — configure maxSurge and maxUnavailable for fine-grained control
$ kubectl set image deployment/myapp nginx=nginx:1.25

Safety Net
Rollback — Instant Recovery
↩️
Revert to previous version instantly — one command sends you back to a working state
🛡
Application stays stable — if v2 has bugs, users are never stuck. v1 comes back immediately.
📜
Deployment history is kept — K8s stores old ReplicaSets so you can roll back to any previous revision
🚨
Critical for production systems — bad releases happen. Rollback is your safety net every time.
$ kubectl rollout undo deployment/myapp
$ kubectl rollout history deployment/myapp

06 / 09
Scaling
Scaling Applications with Deployment
📈
Scale up instantly — increase replicas, Deployment updates its ReplicaSet which creates new Pods
📉
Scale down to save resources — reduce replicas, excess Pods are gracefully terminated
🚦
Respond to traffic changes — scale out during peak hours, scale in overnight
🤖
Works with HPA — Horizontal Pod Autoscaler automatically adjusts replicas based on CPU/memory metrics
Scale up
$ kubectl scale --replicas=6 deploy/myapp
