The Problem
Why Regular Volumes Aren't Enough
⚡
Containers are ephemeral — they start, run, and die. Their filesystem is wiped on every restart.
🔗
Basic volumes are tied to the Pod — emptyDir disappears when the Pod is deleted. Not truly persistent.
🚚
Pods can move across nodes — Kubernetes may reschedule a Pod to a different node, leaving data behind
🏭
Production apps need independent storage — databases cannot rely on storage that's destroyed with the Pod
🔄
Storage must outlive Pods — if a new Pod replaces an old one, it must connect to the same existing data
Key gap: We need storage that lives outside the Pod lifecycle — that's exactly what Persistent Volumes solve.

02 / 09
Definition
What is a Persistent Volume?
🗄
A storage resource in the Kubernetes cluster — provisioned by an admin or dynamically by Kubernetes itself
🔓
Exists independently from Pods — PV has its own lifecycle. Pod dies, PV survives. Pod restarts, data is still there.
⚙
Managed by the Kubernetes cluster — K8s tracks PV state (Available, Bound, Released). You don't manage disks manually.
♻️
Can be reused by applications — when one application is done, the PV can be made available again for another
🌐
Backed by real storage — cloud disk, NFS server, local SSD — PV is an abstraction over real physical or cloud storage
Simple analogy: PV = a permanent room in the building 🏢. Tenants (Pods) may come and go, but the room always stays.

End-to-End Flow
How Persistent Volume Works — Step by Step
1
Admin creates a Persistent Volume
PV is defined (size, type, access mode) — e.g. 50GB AWS EBS disk. Exists in cluster, status: Available
👨‍💼
↓
2
Application requests storage via PVC
App creates a PersistentVolumeClaim (PVC) — says "I need 20GB storage". K8s finds a matching PV.
📋
↓
3
Kubernetes binds PVC to PV
K8s matches the claim to a suitable PV. PV status changes to Bound. Pod can now use the storage.
🔗
↓
4
Pod mounts the PVC and uses storage
Pod accesses /data path. Reads and writes data to the PV. Database stores records safely.
✏️
↓
5
Pod restarts — PV data is untouched
New Pod starts, re-mounts same PV via PVC — all data is intact. Zero data loss.
✅

Capabilities
Key Features of Persistent Volumes
🛡
Persistent data storage — data survives Pod deletions, crashes, and node rescheduling. Always available.
🔓
Independent from containers — PV has its own lifecycle. Deleting a Pod never deletes the PV or its data.
☁
Supports many storage backends — local disk, NFS, AWS EBS, GCP Persistent Disk, Azure Disk, Ceph, and more
♻️
Reusable across applications — PV can be retained and reclaimed after one app is done, then rebound to another
🏭
Essential for stateful applications — databases, queues, and file stores require PVs to function reliably in production
Access modes: RWO (ReadWriteOnce) · ROX (ReadOnlyMany) · RWX (ReadWriteMany) — controls how many Pods can use a PV simultaneously.

Storage Backends
Types of Storage Backends for Persistent Volumes
💿
Local Disk
Node's own storage
Direct storage on the node machine. Fast but tied to a specific node — if Pod moves, cannot follow.
⚡ Fast · ⚠ Node-specific
🌐
NFS
Network File System
Shared network storage accessible by multiple Pods across different nodes simultaneously.
🤝 Shareable · 🌐 Network-based
☁️
AWS EBS
Amazon Elastic Block Store
☁ Cloud
AWS-managed block storage. Durable, high-performance, automatically replicated within an availability zone.
🔒 Durable · ⚡ High IOPS
🔷
Azure Disk
Microsoft Azure storage
Azure-managed block storage for AKS clusters. Supports Standard HDD, Premium SSD, and Ultra Disk tiers.
🔷 AKS-native · 📈 Scalable
🟢
GCP Persistent Disk
Google Cloud storage
Google-managed block storage for GKE clusters. Redundant, auto-encrypted, and globally available.
🌍 Global · 🔐 Auto-encrypted
🗄
Ceph / Other
Self-hosted distributed storage
Open-source distributed storage (Ceph, GlusterFS, Longhorn). On-premise or hybrid cloud setups.
🏠 On-prem · 🔀 Distributed

Lifecycle
PV Lifecycle — How Storage is Managed Over Time
1
Provisioning
Admin creates PV manually (static) or Kubernetes creates it automatically via StorageClass (dynamic). PV becomes Available.
Status: Available 🟢
2
Binding
Application creates a PVC requesting storage. Kubernetes matches PVC to a suitable PV and binds them together. PV becomes Bound.
Status: Bound 🔵
3
Using
Pod mounts the PVC and actively uses the PV. Data is read and written. Pod can restart multiple times — PV stays bound and data is preserved.
Data: Intact across restarts ✅
4
Reclaiming
When PVC is deleted, reclaim policy decides what happens: Retain (keep data), Delete (delete disk), or Recycle (wipe and reuse).
Retain · Delete · Recycle ♻️
