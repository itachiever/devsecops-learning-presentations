Definition
What is a Multi-Container Pod?
📦
A Pod with more than one container — they run together on the same node, always co-located
🤝
Containers run as a single unit — they are started and stopped together, never split
🎯
Designed to support ONE main application — not two separate apps running side-by-side
🔧
Helper containers support the main app — logging, monitoring, config loading, data sync
⚠️ Common mistake: Don't put two separate applications in one Pod. They should be tightly coupled to one purpose.
✅ Correct — one purpose

The Reason
Why Use Multi-Container Pods?
🤝
Containers work together very closely — too tightly coupled to be in separate Pods
🌐
Share the same network (one IP) — talk via localhost with zero latency — no network overhead
💾
Share volumes (storage) — one container writes a file, another reads it instantly — no copying needed
⚡
Extremely fast & efficient communication — localhost is faster than any network call between Pods
🔧
Offload helper tasks — keep your main app clean; let sidecars handle logging, metrics, config

Communication
How Containers Talk Inside a Pod
🏠
Same Pod = same localhost — containers communicate using localhost:port — like programs on the same machine
⚡
Fastest possible communication — no network hop, no DNS lookup, no overhead. Pure speed.
💾
Share volumes for data exchange — one container writes a file, the other reads it. Zero copying needed.
🌐
Different ports within same IP — Main app on :3000, sidecar on :8080, monitor on :9090
Simple rule: Inside Pod → localhost  |  Between Pods → Pod IP or Service name

Important Notes
Things to Know & Watch Out For
🔗
All containers share the same lifecycle — they are created, started, and stopped together as one unit
🔁
Pod restarts = all containers restart — if the Pod is killed, every container inside starts fresh together
🎯
Keep containers tightly related — they should all serve a single application purpose, not multiple ones
✅
Use init containers for setup tasks — init containers run first and finish before the main app starts
⚠️
Avoid unrelated applications — a web server and a database should NOT be in the same Pod
