Why It Matters
Why Troubleshooting is Critical
💥
Applications fail unexpectedly in production — a bad config update, an OOM kill, a missing secret. Failures happen. The question is how fast you find and fix them.
📦
Pods crash, restart, or get stuck — CrashLoopBackOff, Pending, OOMKilled, Evicted. Each state tells a different story. You must know how to read them.
🌐
Networking & resource issues are invisible — a service not routing traffic, DNS failing silently, a Pod unable to reach another — these don't always throw obvious errors.
⚡
Fast troubleshooting = less downtime — every minute of production downtime has a cost. Engineers who troubleshoot systematically resolve issues 10× faster than those who guess.
🎯
Top interview skill for DevOps/DevSecOps — "How would you debug a CrashLoopBackOff?" is one of the most common Kubernetes interview questions. Know the answer cold.
Analogy: Troubleshooting = being a doctor 🏥. You don't guess diagnoses. You check symptoms, run tests, form a hypothesis, apply treatment, and verify it worked. Systematic, not random.

Common Errors
Common Kubernetes Issues
💥
CrashLoopBackOff
Container keeps crashing
Container starts, crashes, Kubernetes restarts it, it crashes again. Exponential backoff delay grows: 10s → 20s → 40s → 5min.
❌ App error / unhandled exception
❌ Missing env vars or secrets
❌ Failed liveness probe
kubectl logs <pod> --previous
🚫
ImagePullBackOff
Container image not found
Kubernetes can't pull the container image. Stuck at init, never runs. Node has no container to start.
❌ Wrong image name or tag
❌ Private registry, no pull secret
❌ Registry unreachable from node
kubectl describe pod <pod>
⏳
Pending Pod
Scheduler can't place Pod
Pod created but not scheduled to any node. Stays Pending forever. Scheduler has rejected all nodes.
❌ Insufficient CPU or memory
❌ No node matches node selector
❌ PVC not bound to storage
kubectl describe pod → Events
🧠
OOMKilled & Network
Memory & connectivity
OOMKilled = Out Of Memory — container hit its memory limit and was killed by the kernel. Network = Pods can't reach each other or external services.
❌ Memory limit too low
❌ Memory leak in app
❌ NetworkPolicy blocking traffic
kubectl top pod · kubectl exec

Essential Commands
Important kubectl Commands
📋
kubectl get pods -A — list all Pods across all namespaces. Quickly spot anything not in Running state. Add -o wide to see node assignment.
🔍
kubectl describe pod <name> — your most powerful debug tool. Shows full spec, conditions, and the Events section with exactly what Kubernetes did and why it failed.
📜
kubectl logs <pod> -c <container> — read container stdout/stderr. Use --previous for crashed containers, -f to stream live.
💻
kubectl exec -it <pod> -- bash — open an interactive shell inside a running container. Debug network, check files, test connections from inside the Pod.
📊
kubectl top pods / kubectl top nodes — see real-time CPU and memory usage. Requires Metrics Server. Spot which Pod is consuming excessive resources instantly.
Command Reference
$ kubectl get pods -A
→ List all pods, all namespaces
$ kubectl describe pod my-pod
→ Full details + events
$ kubectl logs my-pod --previous
→ Logs from crashed container
$ kubectl exec -it my-pod -- bash
→ Shell inside container
$ kubectl top pods --sort-by=cpu
→ CPU usage, sorted
$ kubectl get events --sort-by=.lastTimestamp
→ Recent cluster events
$ kubectl rollout status deploy/app
→ Deployment rollout health
💡 Start with get → describe → logs

Pod Debugging
Checking Pod & Container Issues
📊
Check status & restart count — kubectl get pod my-pod. High RESTARTS = crash loop. STATUS column: Running, Pending, Terminating, OOMKilled, Error.
📋
Describe for events — kubectl describe pod my-pod. Scroll to Events: section. Shows "Failed to pull image", "OOM killed", "Insufficient cpu", "Back-off restarting".
📜
Read application logs — kubectl logs my-pod --previous. The crash reason is almost always in the last few lines. Look for ERROR, FATAL, panic, exception.
💻
Exec into the container — kubectl exec -it my-pod -- sh. Check env vars, test DB connection with curl or nc, inspect files.
CrashLoop debug recipe: get → describe → logs --previous → fix env/secret → redeploy → verify Running.

Network Debugging
Network & Service Troubleshooting
🌐
Verify Service config — kubectl get svc and kubectl describe svc my-svc. Check that the selector labels on the Service match the Pod labels exactly — one typo breaks routing.
🔤
Test DNS resolution — exec into a Pod and run nslookup my-svc.my-namespace.svc.cluster.local. If DNS fails, check CoreDNS Pod health first.
📡
Test Pod-to-Pod connectivity — exec into Pod A and curl <Pod-B-IP>:8080. If it fails, check NetworkPolicy. If it works but svc fails, check selector.
🔀
Check Endpoints — kubectl get endpoints my-svc. If the endpoints list is empty, your Service selector doesn't match any running Pod. Classic mislabel bug.
🚧
NetworkPolicy blocking? — kubectl get networkpolicy -A. A deny-all policy is a common cause of silent connection failures. Deploy a debug Pod with nicolaka/netshoot image.

Resource & Node Debugging
Resource & Node Troubleshooting
📊
Check real-time usage — kubectl top nodes and kubectl top pods --sort-by=memory. Identify which node or Pod is eating resources.
🖥
Inspect node health — kubectl get nodes and kubectl describe node <node>. Check Conditions: Ready, MemoryPressure, DiskPressure, PIDPressure. Any "True" on pressure = problem.
📦
Scheduling failures — Pending Pods — kubectl describe pod Events section shows "Insufficient cpu", "Insufficient memory", "No nodes available". Scale node group or reduce requests.
⚙
Check resource requests vs limits — kubectl get pod -o yaml | grep -A4 resources. Pods without requests cause poor scheduling. Pods without limits cause OOMKills.
Rule of thumb: Always set requests (for the scheduler) AND limits (for protection). Missing either causes subtle, hard-to-debug failures.

Best Practices
Troubleshooting Best Practices
🔬
Always troubleshoot systematically — resist the urge to randomly restart things or apply changes you don't understand. Follow: observe → hypothesize → test → verify. One change at a time.
📊
Monitor continuously — don't wait for failures — Prometheus alerts on CPU spike, memory trend, error rate. Catch issues in early stages before they cascade into outages.
📋
Use centralized logging & monitoring — kubectl logs isn't enough in production. EFK/Loki + Prometheus/Grafana give you the full picture across all Pods, all time.
📖
Document every incident — write a postmortem: what broke, how it was found, how it was fixed, how to prevent recurrence. Your future self will thank you at 3 AM.
🤖
Automate alerts & runbooks — every known failure mode should have an automated alert and a step-by-step runbook. Reduce cognitive load on on-call engineers.
Golden rule: get → describe → logs → exec. This sequence solves 90% of Kubernetes issues. Master these four commands and you'll handle almost anything production throws at you.
✅ The Debug Checklist
□kubectl get pods — check STATUS
□kubectl describe pod — read Events
□kubectl logs --previous — crash reason
□kubectl top pods — resource usage
□kubectl get events — cluster events
□kubectl get endpoints — svc routing
□kubectl exec -it -- bash — test inside

✗Changing multiple things at once
✗Skipping the Events section
✗Not documenting what fixed it
