The Problem
Why Default Kubernetes is Wide Open
🌐
All Pods can talk to all Pods by default — no restrictions. A frontend Pod can directly reach the production database. A logging Pod can hit the payment service.
💀
Lateral movement becomes trivial — if one Pod is compromised, an attacker can use it to reach any other Pod in any namespace across the whole cluster
🗄
Databases exposed to all workloads — without policies, every Pod in the cluster can attempt a connection to your production database. Huge security risk.
📡
Data exfiltration is easier — a compromised Pod can freely communicate with external IPs to leak data, because egress is unrestricted by default
🛡
Network Policies = cluster firewall — apply rules that say exactly which Pods can talk to which, and block everything else. Micro-segmentation.
Default state: Kubernetes is a flat network. Every Pod can reach every other Pod. Network Policies are the only way to change that.

Definition
What is a Network Policy?
📋
A Kubernetes object defining traffic rules — specifies which Pods can send traffic to (ingress) or receive traffic from (egress) other Pods or external IPs
🔥
Works like a firewall for Pods — except instead of IP-based rules, it uses labels and selectors to identify which workloads the rules apply to
🏷
Uses label selectors to target Pods — policies select Pods using podSelector labels. Flexible, dynamic, and cloud-native.
➕
Additive — multiple policies combine — if a Pod matches multiple NetworkPolicies, all their rules are applied. Union of all allowed traffic.
🚫
Default deny when any policy exists — once a NetworkPolicy selects a Pod, all traffic NOT explicitly allowed is automatically blocked
Analogy: NetworkPolicy = security guard at each room 🚪. They check ID (labels) before letting anyone in or out. No badge = no entry.
NetworkPolicy YAML
kind: NetworkPolicy
spec:
 podSelector:
  matchLabels:
    app: database
 policyTypes:
  - Ingress
 ingress:
 - from:
   - podSelector:
      matchLabels:
        app: backend
✅ Only backend → database allowed

Label-Based Targeting
Pod Selectors & Label Matching
🏷
NetworkPolicies use labels to select Pods — rather than IP addresses, Kubernetes uses label selectors. Dynamic and portable across deployments.
🎯
podSelector targets the protected Pod — spec.podSelector selects WHICH Pods the policy applies to (the target)
🔍
from.podSelector identifies allowed sources — in ingress rules, from.podSelector defines WHICH Pods are allowed to send traffic
🌐
namespaceSelector for cross-namespace — namespaceSelector allows or blocks traffic from specific namespaces
📡
ipBlock for external IP ranges — ipBlock.cidr allows traffic from specific external IP ranges. Useful for ingress controllers or external APIs.
Key insight: Labels make policies resilient — if a Pod is replaced or rescheduled, the policy automatically applies to the new Pod as long as it has the same labels.
Label Selector Types
📦 podSelector
app: backend → targets backend Pods
🌐 namespaceSelector
env: prod → all Pods in prod ns
📡 ipBlock
cidr: 10.0.0.0/24 → IP range
🎯 Combined selector
podSelector AND namespaceSelector used together = more precise targeting

Benefits
Why Use Network Policies?
🛡
Improves cluster security dramatically — reduces the blast radius of any breach. A compromised Pod cannot reach databases, secrets services, or payment APIs.
🚫
Blocks lateral movement — in a flat network, attackers move freely between services. Network Policies stop that cold. Attackers stay trapped in one Pod.
🔬
Enables micro-segmentation — treat each service as its own security zone. Frontend, backend, database — each zone with its own strict access rules.
📋
Compliance and audit-friendly — regulations like PCI-DSS, HIPAA require network segmentation. Network Policies provide it natively in Kubernetes.
🔄
Dynamic and label-driven — policies auto-apply when new Pods with matching labels are deployed. No manual firewall rule updates needed.
Best practice: Apply a default-deny-all policy to every namespace. Then explicitly allow only the connections each service actually needs. Zero-trust networking.
Default Deny Pattern
# Block ALL ingress & egress:
kind: NetworkPolicy
spec:
 podSelector: {} # ALL pods
 policyTypes:
  - Ingress
  - Egress
 # no rules = deny all
❌ All traffic blocked by default
✅ Add allow rules one by one
🛡 Zero-trust network model

Important Requirement
Network Policies Require a CNI Plugin
⚠
NetworkPolicy is just a spec — CNI does the enforcement — without a compatible CNI, NetworkPolicy objects are created but have zero effect. Traffic flows freely.
🔧
Built-in flannel does NOT support NetworkPolicy — many default cluster setups use flannel which ignores policies entirely. You need to switch to a policy-capable CNI.
✅
Choose a CNI that supports policies — Calico, Cilium, Weave, Antrea all enforce NetworkPolicies. Pick one before you rely on them for security.
📋
Verify your CNI before deploying policies — run kubectl describe pod -n kube-system to check which CNI is installed
🦁
Calico
Most popular. Full NetworkPolicy + CalicoNetworkPolicy (extended). eBPF or iptables. Great for production.
⭐ Most popular
🐝
Cilium
eBPF-powered. L7 visibility (HTTP/gRPC aware). Hubble for observability. Best for advanced use cases.
🚀 Most advanced
🕸
Weave
Simple setup. Good for smaller clusters. Supports NetworkPolicy. Less complex than Calico/Cilium.
🎯 Simple to use
🐜
Antrea
VMware-backed. Open vSwitch under the hood. Good enterprise support, Kubernetes-native design.
🏭 Enterprise grade
