Dockerfile

A Beginner's Guide

🤔 What is a Dockerfile?

A Dockerfile is a text file that contains step-by-step instructions to build a Docker image. Think of it as a recipe that tells Docker how to create your application container.

🍰 Recipe Analogy

Dockerfile = Recipe
Just like a recipe tells you what ingredients to use and what steps to follow to make a cake, a Dockerfile tells Docker what base image to use and what steps to follow to build your application.

Key Point: Dockerfile creates consistent, reproducible environments for your applications!

🎯 Why Use Dockerfiles?

Consistency: Same environment everywhere - development, testing, production
Automation: No manual setup - Docker builds everything automatically
Version Control: Track changes to your environment setup
Sharing: Team members get identical development environments
Portability: Run anywhere Docker is installed

Base Image & Working Directory

🏠

FROM - Choose Your Foundation

The FROM instruction specifies the base image - like choosing what type of house foundation you want to build on.

# Popular base images: FROM ubuntu:20.04 FROM node:18 FROM python:3.9 FROM nginx:alpine

🎯 Official Images

Use official images from Docker Hub for reliability and security.

🏔️ Alpine Images

Smaller, lightweight versions (e.g., node:alpine) for production.

📁

WORKDIR - Set Your Workspace

The WORKDIR instruction sets the working directory inside the container - like choosing which room in your house to work in.

# Set working directory WORKDIR /app # All subsequent commands run from /app WORKDIR /usr/src/app WORKDIR /var/www/html

📍 Best Practice

Always set WORKDIR to avoid running commands in random directories.

🔄 Multiple WORKDIR

You can use WORKDIR multiple times to change directories.

Copying Files & Running Commands

📋

COPY - Add Your Ingredients

The COPY instruction copies files from your computer (host) into the container - like adding ingredients to your kitchen.

# Copy specific file COPY package.json /app/ # Copy entire directory COPY . /app/ # Copy with rename COPY config.txt /app/settings.txt
Source: File/folder on your computer
Destination: Path inside container
Tip: Copy package.json first, then source code for better caching
⚙️

RUN - Execute Commands

The RUN instruction executes commands inside the container during build time - like following cooking steps in your recipe.

# Install packages RUN apt-get update && apt-get install -y curl # Install Node.js dependencies RUN npm install # Build application RUN npm run build
Build Time: RUN commands execute when building the image
Layers: Each RUN creates a new layer in the image
Tip: Combine commands with && to reduce layers

Exposing Ports & Default Commands

🚪

EXPOSE - Open the Door

The EXPOSE instruction tells Docker which ports your application uses - like putting a door number on your restaurant.

# Expose web server port EXPOSE 80 # Expose Node.js app port EXPOSE 3000 # Expose multiple ports EXPOSE 8080 8443

🔍 Important Note

EXPOSE is documentation - it doesn't actually publish ports. Use docker run -p to publish ports to the host.

🚀

CMD & ENTRYPOINT - Start Serving

CMD: Default command when container starts

ENTRYPOINT: Fixed command that always runs

# CMD - can be overridden CMD ["npm", "start"] CMD ["python", "app.py"] # ENTRYPOINT - always runs ENTRYPOINT ["python", "app.py"] # Combined usage ENTRYPOINT ["python"] CMD ["app.py"]
CMD: Can be overridden when running container
ENTRYPOINT: Cannot be overridden, always executes
Best Practice: Use CMD for default behavior

Key Difference: CMD can be overridden, ENTRYPOINT always runs. Use CMD for flexibility!

Complete Dockerfile Example

Here's a real Dockerfile for a Node.js web application with detailed explanations:

📄 Sample Dockerfile

# Use official Node.js runtime as base image FROM node:18-alpine # Set working directory inside container WORKDIR /usr/src/app # Copy package files first (for better caching) COPY package*.json ./ # Install dependencies RUN npm ci --only=production # Copy application source code COPY . . # Create non-root user for security RUN addgroup -g 1001 -S nodejs RUN adduser -S nextjs -u 1001 # Change to non-root user USER nextjs # Expose port that app runs on EXPOSE 3000 # Command to start the application CMD ["npm", "start"]

🔍 Step-by-Step Breakdown

Line 2: Start with Node.js 18 Alpine (lightweight Linux)
Line 5: Set /usr/src/app as our workspace
Line 8: Copy only package files first
Line 11: Install production dependencies
Line 14: Copy all application source code
Lines 17-18: Create secure non-root user
Line 21: Switch to non-root user
Line 24: Document that app uses port 3000
Line 27: Start the application with npm start

⚡ Build & Run Commands

# Build image from Dockerfile docker build -t my-node-app . # Run container from image docker run -p 3000:3000 my-node-app # Visit http://localhost:3000

🎯 Best Practices Applied

📦 Layer Optimization

Copy package.json first, then install dependencies. This allows Docker to cache the dependency installation layer.

🔒 Security

Create and use a non-root user to run the application, reducing security risks.

📏 Size Optimization

Use Alpine Linux base image (smaller) and install only production dependencies.

📖 Documentation

Use comments to explain each step, making the Dockerfile self-documenting.
