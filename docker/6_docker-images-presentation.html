🐳 What is a Docker Image?

📋 Definition

  • A Docker image is a lightweight, standalone package that contains everything needed to run an application
  • Includes code, runtime, libraries, environment variables, and configuration files
  • Images are read-only templates used to create containers
  • Foundation of containerization and modern DevOps workflows

🔑 Key Benefits

  • Consistent environments across development, testing, and production
  • Faster deployment and scaling
  • Improved resource utilization
🏗️

Think of it like a Blueprint

Just like an architectural blueprint contains all the instructions to build a house, a Docker image contains all the instructions and components needed to create and run a container.


Blueprint → House
Docker Image → Container

🏗️ Layered Structure

📚 How Layers Work

  • Base Image: Foundation layer (e.g., Ubuntu, Alpine Linux)
  • Application Layers: Each command in Dockerfile creates a new layer
  • Read-Only: All layers are immutable once created
  • Efficient Storage: Layers are shared between images
  • Incremental Updates: Only changed layers need to be downloaded

💡 Why Layers Matter

If you update your application code, only the top layer needs to be rebuilt and redistributed - not the entire image!

Layer Stack Example

🐧 Base: Ubuntu 20.04
☕ Install: Java Runtime
📦 Install: Dependencies
📁 Copy: Application Code
⚙️ Configure: Environment
🚀 Entrypoint: Start App

Each layer builds upon the previous one

🔨 Building Images

📋 Dockerfile Basics

  • Dockerfile: Text file with instructions to build an image
  • Commands: Each line creates a new layer
  • Build Process: Docker reads Dockerfile top to bottom
  • Best Practice: Order commands by frequency of change

🛠️ Common Build Commands

$ docker build -t myapp:1.0 .
$ docker build -t myapp:latest --no-cache .

The . tells Docker to look for Dockerfile in current directory

Sample Dockerfile

FROM node:16-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["npm", "start"]
Layer Optimization: Dependencies are copied and installed before application code, enabling layer caching.

🏷️ Versioning & Tags

🔖 What are Tags?

  • Human-readable labels for image versions
  • Format: repository:tag (e.g., nginx:1.21)
  • Default tag: "latest" if no tag specified
  • Multiple tags: Same image can have multiple tags
  • Best Practice: Use semantic versioning

📊 Tagging Strategies

  • Semantic: v1.2.3, v1.2, v1
  • Environment: dev, staging, prod
  • Feature: feature-auth, hotfix-security
  • Timestamp: 2025-01-15

Tag Examples

nginx:latest
nginx:1.21
myapp:v2.1.0
myapp:dev
redis:6-alpine

⚠️ Interview Tip

Never use "latest" in production! It's unpredictable and can break your application when the image updates.

🛠️ Tagging Commands

$ docker tag myapp:latest myapp:v1.0
$ docker build -t myapp:v1.1.0 .

🏪 Registry Types

  • Docker Hub: Public registry (free tier available)
  • AWS ECR: Amazon's private registry
  • Google GCR: Google Cloud Registry
  • Azure ACR: Azure Container Registry
  • Harbor: Open-source private registry
  • Private: Self-hosted registries

🛠️ Essential Commands

$ docker login
$ docker push myapp:v1.0
$ docker pull nginx:latest
$ docker search ubuntu
💡 Pro Tip: Always use specific tags when pulling images in production environments for consistency.

🎯 Practical Example & Interview Tips

🚀 Real-World Example: Node.js App

# Build the image
$ docker build -t webapp:v1.0 .
# Run locally for testing
$ docker run -p 3000:3000 webapp:v1.0
# Tag for production
$ docker tag webapp:v1.0 myregistry/webapp:v1.0
# Push to registry
$ docker push myregistry/webapp:v1.0

📋 Image Inspection

$ docker images
$ docker history webapp:v1.0
$ docker inspect webapp:v1.0
