🐳 What is a .dockerignore File?

1/6

.dockerignore is a special file that tells Docker which files and folders to ignore when building an image.

📝 Think of it like .gitignore for Docker!

Just like .gitignore tells Git what not to track, .dockerignore tells Docker what not to include in your image build process.

🎯 What it does:

  • • Excludes unnecessary files from Docker build context
  • • Prevents sensitive files from entering your image
  • • Speeds up build process significantly
  • • Reduces final image size

⚡ Why it matters:

  • • Without it, Docker copies EVERYTHING
  • • Builds become slow and bloated
  • • Security risks from exposed files
  • • Wasted storage and bandwidth
Key Point: The .dockerignore file must be placed in the same directory as your Dockerfile (usually project root).

🚀 Why Use .dockerignore?

2/6

📦 Smaller Images

Exclude unnecessary files like documentation, tests, and development tools. A typical Node.js app can shrink from 500MB to 150MB!

⚡ Faster Builds

Less data to transfer means faster build context upload. Builds that took 2 minutes can complete in 20 seconds.

🔒 Better Security

Keep sensitive files (.env, passwords, private keys) out of your images. No accidental exposure of secrets!

🧹 Cleaner Images

Remove development clutter like logs, temporary files, and cache directories. Production images stay lean and focused.

📊 Real Impact Example:

WITHOUT
🐌 3.2GB image
⏱️ 5min builds
→
WITH
🚀 800MB image

⚠️ Common Mistake:

Forgetting .dockerignore means your entire project directory (including node_modules, .git, logs) gets copied into every Docker layer, making images huge and slow!

📍 File Location

my-project/ ├── Dockerfile ├── .dockerignore ← Here! ├── src/ ├── tests/ └── README.md

Important: Must be in same directory as Dockerfile!

🔄 Build Context Process

1. Docker runs: docker build .

2. Reads .dockerignore rules

3. Filters out matching files

4. Sends clean context to daemon

💡 Key Concept: Build Context

Build context is all the files Docker considers for your build. By default, it's everything in the folder where you run docker build. .dockerignore helps you control what gets included!

📝 Syntax & Patterns

4/6

🔤 Basic Syntax

# Comments start with # *.log # All .log files temp/ # Entire temp directory node_modules # node_modules folder !important.log # Exception: keep this file

🎯 Wildcards

* # Any file *.txt # All .txt files test* # Files starting with "test" **/*.log # .log files in any subdirectory file.? # file.a, file.b, etc.

⚠️ Common Patterns

# Development files .git .env *.env.* # Dependencies node_modules vendor/ # Build artifacts dist/ build/ *.tmp # IDE files .vscode/ *.swp

❗ Exceptions

# Ignore all .env files *.env # But keep the template !.env.example

🚨 Important Rules:

  • Patterns are relative to build context root
  • Exceptions (!) must come after the ignore rule
  • Directory patterns should end with / for clarity
  • One pattern per line

🔒 Security Best Practices

❌ Always Ignore:

  • .env files with secrets
  • Private keys (.pem, .key)
  • Database files
  • Personal credentials

✅ Safe to Include:

  • Source code
  • Configuration templates
  • Public assets
  • Package definitions

🎤 Interview & Best Practices

6/6

❓ Common Interview Questions

Q: "What happens if you don't use .dockerignore?"

A: Large images, slow builds, security risks

Q: "Where does .dockerignore go?"

A: Same directory as Dockerfile

Q: "Can you have multiple .dockerignore files?"

A: No, only one per build context

🎯 Pro Tips

  • 🔍 Test it: Use docker build --no-cache to verify
  • 📋 Start broad: Ignore categories, then add exceptions
  • 🔄 Version control: Commit .dockerignore to git
  • 📖 Document: Comment complex patterns
