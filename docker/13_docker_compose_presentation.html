What is a Docker Compose File?

Simple definition, purpose & multi-container setup in DevOps

📋 Definition

A Docker Compose file is a YAML configuration file that defines and manages multiple Docker containers as a single application stack.

🎯 Why Multi-Container?

  • Real applications have multiple components
  • Database + Web App + Cache + Load Balancer
  • Each service runs in its own container
  • Easy to scale individual components

Traditional vs Docker Compose

Before: Manual container management
docker run -d nginx
docker run -d mysql
docker run -d redis
With Compose:
docker compose up
→ Starts all services together!

🚀 DevOps Benefits

Consistent environments across development, testing, and production. One command to rule them all!

Core Structure

YAML format & main keywords: version, services, networks, volumes

📝 YAML Basics

  • Indentation matters! Use 2 spaces
  • Key-value pairs: key: value
  • Lists use - (dash + space)
  • Comments start with #

🔑 Main Keywords

version: Compose file format version

services: Define your containers

networks: Custom networks for communication

volumes: Persistent data storage

📄 Basic Structure

version: "3.8" services: web: image: nginx ports: - "80:80" database: image: mysql environment: - MYSQL_ROOT_PASSWORD=secret networks: default: driver: bridge volumes: db_data: driver: local

💡 Remember

File must be named docker-compose.yml or compose.yml

Service Configuration

Define containers: image, build, ports, environment, depends_on

🔧 Service Options

image: Use pre-built image from registry

build: Build from Dockerfile

ports: Map container ports to host

environment: Set environment variables

depends_on: Start services in order

volumes: Mount files/directories

🌐 Port Mapping

ports: - "8080:80" # host:container - "3306:3306"

Access on localhost:8080 → goes to container:80

📋 Complete Service Example

services: web: build: ./app # Build from Dockerfile ports: - "3000:3000" environment: - NODE_ENV=development - DB_HOST=database depends_on: - database # Start DB first volumes: - ./app:/usr/src/app database: image: postgres:13 # Use pre-built image environment: - POSTGRES_DB=myapp - POSTGRES_USER=admin - POSTGRES_PASSWORD=secret volumes: - db_data:/var/lib/postgresql/data ports: - "5432:5432" volumes: db_data:

Volumes and Networks

Persistent data storage & inter-container communication

💾 Volumes (Data Persistence)

  • Named volumes: Docker manages storage
  • Bind mounts: Map host directories
  • Tmpfs mounts: Temporary in-memory storage
  • Data survives container restart/removal
volumes: # Named volume db_data: driver: local services: database: volumes: - db_data:/var/lib/mysql # Named - ./config:/etc/mysql # Bind mount

⚠️ Important

Without volumes, data is lost when container stops!

🌐 Networks (Communication)

Frontend (web)
Backend (api)
Database (db)
  • Services can talk using service names
  • Default network created automatically
  • Custom networks for isolation
networks: frontend: driver: bridge backend: driver: bridge services: web: networks: - frontend api: networks: - frontend - backend db: networks: - backend

Workflow & Commands

Essential commands & DevOps team workflow

⚡ Essential Commands

$ docker compose up

Start all services

$ docker compose up -d

Start in background (detached)

$ docker compose down

Stop and remove containers

$ docker compose logs

View logs from all services

$ docker compose ps

List running services

$ docker compose exec web bash

Access container shell

🔄 DevOps Workflow

1. Development

  • Developer writes docker-compose.yml
  • Commits to version control (Git)
  • Team members can replicate environment

2. Team Collaboration

  • Sharing: Same file across team
  • Environment parity: Dev = Staging = Prod
  • Onboarding: New developers get started quickly

💡 Best Practices

  • Use specific image tags (not :latest)
  • Define health checks for services
  • Use multi-stage builds for optimization
  • Keep secrets in environment files
  • Use networks for service isolation

🎯 Common Interview Questions

Q: What's the difference between docker run and docker compose?

A: docker run manages single containers, Compose manages multi-container applications.

Q: How do containers communicate?

A: Using service names as hostnames within the same network.

Q: What happens to data when containers stop?

A: Lost unless stored in volumes or bind mounts.
