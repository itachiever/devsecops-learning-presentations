🛡️ OWASP Docker Top 10

D05 – Maintain Security Contexts
  • 🎯 Essential protection layer for Docker containers
  • 🔒 Controls what containers can and cannot do
  • ⚠️ Often overlooked but critical for security
2/10

🔐 What is a Security Context?

  • 🛡️ Security boundaries that control container permissions
  • 👤 User/Group IDs: Who the container runs as
  • 🚫 Capabilities: What system functions are allowed
  • 📁 File permissions: What files can be accessed
  • 🔒 System calls: What the container can request from kernel
Think of it as an ID card that says "You can only do THIS"
3/10

🚨 Why Security Contexts Matter

  • 🔥 Container escape prevention: Stops containers from breaking out
  • 👑 Privilege escalation: Prevents gaining root access
  • 🏠 Host protection: Shields your server from container attacks
  • 📊 Compliance: Many regulations require proper contexts
  • 🛡️ Defense in depth: Additional security layer beyond images
4/10

⚙️ How Security Contexts Work

  • 🐧 SELinux: Mandatory access control for Linux
  • 🛡️ AppArmor: Application security profiles
  • 🔒 seccomp: System call filtering
  • 👤 User namespaces: Isolate user IDs
  • 🚫 Capabilities: Fine-grained permissions
  • 📁 Read-only filesystems: Prevent modifications
# Basic security context example
docker run --user 1000:1000 --read-only --cap-drop=ALL myapp
5/10

💻 D05 Examples: Real Commands

# BAD: Running as root with all privileges
docker run -d myapp
# GOOD: Non-root user with dropped capabilities
docker run -d --user 1000:1000 --cap-drop=ALL --cap-add=NET_BIND_SERVICE --read-only --security-opt=no-new-privileges myapp
  • ✅ Always specify non-root user (--user)
  • ✅ Drop all capabilities, add only needed ones
  • ✅ Use read-only filesystem when possible
6/10

🎯 Attack Vectors: What Hackers Try

  • 💀 Container escape: Break out to host system
  • 👑 Privilege escalation: Gain root access from limited user
  • 📁 File system access: Read/write sensitive host files
  • 🔓 Capability abuse: Use excessive permissions for attacks
  • 🐚 Shell injection: Execute commands via weak contexts
Without proper contexts, containers = root access to your host!
7/10

💥 Impact: What Goes Wrong

  • 🔥 Complete host compromise: Attacker owns your server
  • 📊 Data breach: Access to all host data and other containers
  • 💸 Service disruption: Shutdown or modify critical services
  • 🦠 Malware installation: Persistent backdoors on host
  • ⚖️ Compliance violations: Regulatory fines and legal issues
One weak container = Entire infrastructure at risk
8/10

🔍 How to Identify Weak Contexts

# Check running containers
docker ps --format "table {{.Names}}\t{{.RunningFor}}\t{{.Status}}"
# Inspect container security settings
docker inspect CONTAINER_NAME | grep -E "(User|CapDrop|CapAdd|SecurityOpt)"
# Check for root processes
docker exec CONTAINER_NAME ps aux | grep "root"
  • 🚨 Red flags: User: root, no --cap-drop, privileged mode
  • 📋 Use security scanners: Docker Bench, Clair, Trivy
  • 📊 Regular audits: Check all running containers monthly
9/10

🛠️ How to Fix & Prevent

✅ Best Practices:

  • 👤 Always use non-root users
  • 🚫 Drop ALL capabilities first
  • 📖 Use read-only filesystems
  • 🔒 Enable seccomp profiles

❌ Avoid:

  • Never use --privileged
  • Don't run as root (UID 0)
  • Avoid --cap-add=ALL
  • No --security-opt=apparmor:unconfined
10/10

📚 Recap & Key Takeaways

  • 🎯 Security contexts = Container ID cards - Control what containers can do
  • 🚨 Missing contexts = Easy host compromise - Always configure them!
  • 💻 Use --user, --cap-drop=ALL, --read-only - Your security baseline
  • 🔍 Regular audits with docker inspect - Check every container monthly
  • 🛡️ Defense in depth approach - Multiple security layers protect better
Remember: Secure by default, not secure by accident! 🔐
