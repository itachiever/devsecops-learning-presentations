OWASP Docker Top 10
D09 – Follow Immutable Paradigm
  • Secure containers by making them unchangeable after deployment
  • Prevent runtime modifications that create security vulnerabilities
  • Essential practice for production Docker environments
  • Part of OWASP's top security recommendations for containers
🔒
2/10
What Does "Immutable" Mean?
Simple Definition for Containers
  • Immutable = Cannot be changed or modified
  • Container filesystem becomes read-only after startup
  • No files can be added, modified, or deleted inside the container
  • Like a sealed box - you can't alter its contents once closed
Think of it like: A DVD vs. a USB drive
DVD = Immutable (read-only)
USB = Mutable (can write/change files)
📦

Why Immutable Containers Are Secure
Security Benefits
  • Prevents malware installation - attackers can't write files
  • Stops configuration tampering - settings stay unchanged
  • Eliminates drift - containers remain identical to original
  • Simplifies compliance - easier to audit and verify
  • Faster recovery - just restart, no cleanup needed
🛡️
4/10
D09 Examples
Using --read-only and Handling Updates
✅ Good: Read-only container
docker run --read-only nginx:latest

❌ Bad: Writable container
docker run nginx:latest
  • Updates: Build new image → Deploy → Remove old
  • Temporary files: Use tmpfs mounts for /tmp
  • Logs: Write to volumes, not container filesystem

Troubleshooting & Recovery Benefits
How Immutability Simplifies Operations
  • Predictable behavior - containers always start the same way
  • Easy rollback - redeploy previous working image
  • No configuration drift - eliminates "it works on my machine"
  • Fast recovery - restart container instead of debugging
  • Simplified debugging - issues are in code, not runtime changes

Attack Vectors
What Hackers Do with Writable Containers
  • Install backdoors - persistent access tools
  • Plant cryptocurrency miners - steal computing resources
  • Modify system binaries - hide malicious activities
  • Change configurations - disable security features
  • Store stolen data - use container as temporary storage
⚠️ Real Attack: Hackers modify /etc/passwd to create new admin users

Impact of Ignoring Immutability
Consequences of Mutable Containers

Security Risks

  • Data breaches - sensitive information stolen
  • Hidden malware - undetected malicious software
  • Privilege escalation - attackers gain admin access

Operational Issues

  • Configuration drift - inconsistent environments
  • Complex debugging - unknown system state
  • Compliance failures - audit trail lost
8/10
How to Identify Mutable Containers
Detection Methods
  • Check docker run commands - look for missing --read-only
  • Inspect running containers - verify ReadonlyRootfs setting
  • Monitor file changes - detect runtime modifications
  • Review Dockerfile - look for unnecessary write permissions
Command to check:
docker inspect <container> | grep ReadonlyRootfs
# Should return: "ReadonlyRootfs": true
🔍
9/10
How to Fix & Prevent
Steps for Immutable Containers
  • 1Add --read-only flag to docker run commands
  • 2Use tmpfs mounts for temporary files (/tmp, /var/run)
  • 3External volumes for persistent data
  • 4Log to external systems not container filesystem
  • 5Test thoroughly ensure app works read-only
docker run --read-only --tmpfs /tmp nginx:latest

Recap & Best Practices
Key Takeaways

Core Principles

  • Rebuild, don't patch - create new images
  • Data stays outside - use volumes
  • Always --read-only in production

Remember

  • Security by design - prevent, don't detect
  • Consistency matters - same behavior everywhere
  • Simplify operations - restart over repair
